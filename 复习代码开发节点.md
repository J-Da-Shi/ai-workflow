  2. 自问自答（模拟面试）

  3. 最关键的一点
  
  面试不考背代码细节。面试考的是：

  - 你为什么这样设计？（trade-off）
  - 遇到了什么问题，怎么解决的？（实战经验）
  - 有什么不足，怎么改进？（深度思考）

  所以与其记住每行代码，不如记住每个决策背后的 why：
  - 为什么用滑动窗口不用 LLM 摘要？→ 零成本，DeepSeek 场景够用
  - 为什么不用沙盒？→ 快照+还原更轻量，还能复用 Git diff
  - 为什么 SSE 不用 WebSocket？→ 单向推送，SSE 协议更简单
  - 为什么 30 轮改 200 轮？→ 参考 Claude Code 无限制设计
  
  把这些 "为什么" 记住，细节忘了临场看代码就能想起来。

Claude 是怎么进行上下文管理的，我现在的上下文管理是不是太基础了，面试都没法应对？
Claude Code 的做法

  Claude Code 有 200K 上下文窗口，它的策略是 compaction（压缩）：

  - 当对话接近上下文窗口限制时，自动把早期对话内容发给一个小模型生成摘要
  - 用摘要替代原始内容（保留语义，大幅减少 token）
  - 关键文件内容、重要决策、最近操作完整保留

Diff审批，扫描所有文件存快照，如果文件很多很大，是不是会很慢，并且存入的内容也很大，性能堪忧？
 当前问题

  // 扫描整个工作目录，把每个文件内容存进 snapshot
  const snapshot = this.scanDirectory(workDir);
  execution.snapshot = snapshot;  // 存入 MySQL 的 JSON 字段

  如果工作目录是一个 clone 下来的大项目（几千个文件、几百 MB），确实有两个问题：
  - 扫描慢：递归读几千个文件耗时
  - 存储大：snapshot JSON 可能有几十 MB，MySQL 单行存这么大的 JSON 不合理
  
  解决方案（分层）

  方案 A（当前已做的优化）：
  - scanDirectory 已跳过 node_modules / .git / dist
  - 单文件 > 512KB 不读内容
  - 这样实际扫描的文件数量和总大小可控
  
  方案 B（如果仍然太大）—— 只存变更文件的快照：

  不扫描全部文件，而是只对 Agent 实际操作过的文件做快照：

  // 思路：从 agent_logs 中提取 write_file 操作的文件列表
  // 执行前只读取这些文件的内容作为快照（before）
  // 如果是新文件（before 不存在），before 就是空字符串

  这样快照只包含被修改的几个文件，而不是整个项目。

  方案 C（生产级）—— 用 Git 自身的 diff：

  如果配了 Git，根本不需要自己做快照：
  git diff HEAD  # 直接拿到所有变更的 diff
  
  Git 天然就是版本管理工具，它的 diff 比自己扫描快照高效得多。驳回时 git checkout . 一行还原。

# 上下文压缩 — 你的方案 vs 更好的方案
  
  面试回答模板

  ▎ "我当前用的是滑动窗口截断，更好的方案有三个层次："

  你的方案：滑动窗口 + 暴力截断前 200 字符
  - 优点：零成本，零延迟
  - 缺点：截断位置随机，可能丢关键信息
  
  更好的方案 1 — LLM 摘要压缩（Claude Code 用的）：
  - 做法：把早期轮次发给一个小模型（如 Haiku），让它生成一段结构化摘要（"读了哪些文件、改了什么、关键发现"）
  - 优点：语义保留度高
  - 缺点：每次压缩多一次 API 调用（成本 + 延迟）
  - 适合：Claude（200K 窗口，不常触发压缩）
  
  更好的方案 2 — 按重要性分级：
  - 做法：write_file 结果只有一行（"已创建 xxx"），不需要压缩；read_file 结果几百行，是膨胀主因。只对 read_file 的 tool result 做摘要，其他保留
  - 优点：精准控制，不误伤
  
  更好的方案 3 — Token 预算制：
  - 做法：设一个 token 上限（如 8000），每轮调用前计算当前 messages 总 token 数，超了就从最早的开始压缩，直到降到预算内
  - 优点：精确控制，不会超窗口
  - 缺点：需要 tokenizer 计算 token 数
  
  ▎ "我选滑动窗口截断是因为用 DeepSeek（上下文窗口有限、API 调用便宜），不值得为压缩多花一次 API 调用。如果换成 Claude 200K 窗口，我会改用 LLM 摘要方案。"

  # 快照 — 你的方案 vs 更好的方案
  
  面试回答模板

  ▎ "我当前用的是全目录扫描存 JSON，更好的方案有："

  你的方案：scanDirectory 全扫，存到 MySQL JSON 字段
  - 优点：实现简单，不依赖外部工具
  - 缺点：大项目扫描慢，JSON 可能几十 MB
  
  更好的方案 1 — 只快照被操作的文件：
  - 做法：从 Agent 日志中提取 write_file 涉及的文件路径，只对这些文件存 before 内容
  - 优点：精准，数据量小
  - 做法：Agent Loop 开始前不做任何事，结束后遍历 logs 找 write_file 记录，再去读快照
  
  更好的方案 2 — 用 Git diff（最优）：
  - 做法：既然已经有 Git 集成，直接用 git diff HEAD 拿变更，git checkout . 还原
  - 优点：零自研成本，Git 就是做这个的专业工具；diff 格式标准化，前端有成熟渲染库
  - 缺点：依赖 Git（没配 Git 的节点用不了）
  
  更好的方案 3 — 文件系统 Watch：
  - 做法：用 fs.watch 或 chokidar 监听工作目录变化，实时记录哪些文件被改了
  - 优点：不需要全扫，实时感知变更
  - 缺点：有平台兼容性问题，实现复杂
  
  ▎ "我选全目录扫描是因为实现最简单，且通过排除 node_modules/.git + 512KB 大小限制控制了范围。如果项目很大，最优方案是直接用 Git diff——我已经有 Git 集成了，审批时用 
  ▎ git diff 拿变更，驳回时 git checkout . 还原，比自己管理快照更可靠。"
