 1. 画调用链路图（手画）
  
  不看代码，凭记忆画出来。画不出来的地方就是你没真正理解的地方，回去看那段代码。

  比如画这条链路：

  用户点"执行节点"
    → 前端发生了什么？（fetch 哪个接口？怎么读 SSE？）
    → 后端 Controller 做了什么？（设什么 Header？怎么写 SSE？）
    → ExecutionService 做了什么？（快照？Git？调 runAgent？）
    → runAgent 内部做了什么？（循环几步？怎么判断结束？）
    → 工具怎么执行的？（safePath？executeTool switch？）
    → 结果怎么返回前端？（onEvent → res.write → reader.read → CustomEvent）

  画完后对照代码核实，标出你画错的地方。

  2. 自问自答（模拟面试）
  
  给自己提这些问题，不看代码回答。答不上来的就是薄弱点：

  Agent Loop：
  - AI 怎么知道有哪些工具可以用？（tools 参数，JSON Schema 定义）
  - AI 怎么告诉后端要调哪个工具？（返回 tool_calls 数组，包含 name + arguments JSON）
  - 后端怎么把工具结果告诉 AI？（push role='tool' 消息，带 tool_call_id 配对） 
  - 循环什么时候结束？（AI 回复中没有 tool_calls）
  
  SSE：
  - 为什么不用 WebSocket？（单向推送够用，SSE 更简单；WebSocket 是双向的，杀鸡用牛刀）
  - 为什么不用浏览器原生 EventSource？（不支持 POST，不支持自定义 Header 带 Token）
  - 前端怎么知道流结束了？（收到 data: [DONE]）
  
  上下文压缩：
  - 为什么不能删除消息？（API 要求 tool_call 和 tool result 一一配对）
  - 压缩具体压缩的是什么？（role='tool' 的 content 截短，不动消息结构）
  
  Diff 审批：
  - 快照存在哪？（MySQL node_executions 表的 snapshot 字段，JSON 格式）
  - 驳回时怎么还原 "新增" 的文件？（直接 fs.rmSync 删掉）
  - 驳回时怎么还原 "修改" 的文件？（用 snapshot 中的旧内容覆盖写回）
  
  Git：
  - 为什么在审批后才 push，不是执行完就 push？（确保远端只有审核过的代码）
  - Token 怎么用于认证？（拼入 URL：https://{token}@github.com/...）
  - 创建 PR 调的什么接口？（GitHub: POST /repos/{owner}/{repo}/pulls）

  ---
  3. 最关键的一点
  
  面试不考背代码细节。面试考的是：

  - 你为什么这样设计？（trade-off）
  - 遇到了什么问题，怎么解决的？（实战经验）
  - 有什么不足，怎么改进？（深度思考）

  所以与其记住每行代码，不如记住每个决策背后的 why：
  - 为什么用滑动窗口不用 LLM 摘要？→ 零成本，DeepSeek 场景够用
  - 为什么不用沙盒？→ 快照+还原更轻量，还能复用 Git diff
  - 为什么 SSE 不用 WebSocket？→ 单向推送，SSE 协议更简单
  - 为什么 30 轮改 200 轮？→ 参考 Claude Code 无限制设计
  
  把这些 "为什么" 记住，细节忘了临场看代码就能想起来。
