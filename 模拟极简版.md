二、AI Agent 核心实现（加长满分版）
我们的 Agent 整体是基于 ReAct 思想实现的工具调用循环，本质就是一个可控的 While 循环。整体流程是，模型每次思考完成后，如果需要工具辅助就会返回 tool_calls，我前端解析对应的工具类型和参数，去执行读文件、写代码、执行命令这些操作，再把工具执行的结果回填到消息上下文里，进入下一轮循环，直到模型不需要工具、直接输出总结，才算任务结束。
落地过程中我重点解决了两个最核心的工程问题。第一个是上下文膨胀问题，多轮工具调用后消息体量会爆炸，我采用了源头截断加滑动窗口压缩的策略，优先保留最近几轮的完整执行记录，更早的历史轮次做摘要化处理，同时严格保留消息结构和 tool_call 的对应关系，避免模型调用报错。第二个是 Agent 死循环问题，我做了四层防控，包括最大30轮硬性上限、相同参数重复调用检测、连续失败熔断，从工程层面彻底避免无效死循环，保证任务稳定执行。

三、SSE 实时日志（加长满分版）
项目里的 AI 任务都是长耗时的异步操作，需要实时展示执行进度，所以我选用 SSE 做单向流式推送。没有用原生的 EventSource，是因为它不支持 POST 请求和自定义请求头，没法携带 JWT 鉴权，所以我基于 fetch 结合 readableStream 手动封装了一套 SSE 能力。
落地时踩了网络粘包、半包的坑，我通过前端维护缓冲区的方式，持续拼接流式数据，再按照 SSE 双换行符规则切割解析，保证每一条日志事件都是完整的。同时做了完整的异常兜底，包括断线自动重连、超时释放、页面卸载和组件销毁时主动终止请求，彻底解决内存泄漏问题。针对大数量日志卡顿的问题，我结合虚拟滚动只渲染可视区域 DOM，保证上万条日志滚动也能丝滑不卡顿。

八、高频追问满分加长答案
1. 为什么不用 LangChain？
我在开发初期调研过 LangChain，但最终选择原生对接 OpenAI SDK 自研实现。核心原因是 LangChain 的封装太重，内置的 Chain、Memory、Tool 整套抽象概念太多，对于我们固定的代码研发场景来说，大部分能力用不上，反而会增加调试成本和包体积。而且自研的 Agent 循环代码非常精简，可控性极强，每一轮的工具调用、消息回填、异常处理我都可以精准定制，出问题可以快速定位排查，更适合生产环境落地。
2. 为什么选择滑动窗口做上下文压缩？
我对比过三种主流压缩方案，LLM 语义摘要精度高但是成本高、延迟大，还会引入摘要失真的不确定性；精准 Token 预算制精度最好但是实现复杂度高。而滑动窗口是工程性价比最高的方案，规则简单、零延迟、稳定性极强，非常适合做工程兜底。因为 Agent 的决策本身就强依赖最近几步的操作，更早的历史记录只需要保留摘要即可，不需要完整内容，就算丢失部分细节，Agent 也可以通过重新读文件、查代码自行补齐，完全不影响任务执行。
3. 项目的不足和后续优化方向？
当前项目整体链路已经稳定落地，但我梳理了三个核心优化方向。第一，优化文件写入工具，把现在的全量 writeFile 改成增量 editFile，大幅降低 Token 消耗，提升 AI 生成代码的速度。第二，解决单机部署问题，把当前的内存并发锁升级为 Redis 分布式锁，再引入任务队列，支持服务水平扩展。第三，优化 RAG 检索精度，在现有向量检索的基础上增加 Rerank 精排，提升知识库匹配准确率，让 AI 生成代码更贴合团队规范。
4. 你们平台和 Cursor、Copilot 的区别？
核心定位完全不同。Cursor、Copilot 都是面向个人的编码助手，核心是问答式的文本交互，只能辅助写代码。而我们的平台是面向完整研发流程的，核心是工作流自动化。我们可以把 PRD 审核、代码开发、人工审批、Git 提交、PR 创建整条链路串起来，通过多轮 Agent 工具调用完成完整研发任务，还有人工审批闭环兜底，最终交付的是可以直接合入仓库的正式代码，不是简单的文本答案，更适配团队规模化研发提效场景。
