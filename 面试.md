# AI Workflow - 面试口述稿

## 一句话介绍

独立设计并开发了一套 **AI 驱动的软件开发工作流平台**，覆盖 PRD 评审 → 代码生成 → 自测 → Code Review → 部署的全链路 SDLC，核心亮点是内置 Agent Loop 引擎，通过 Tool-Calling 机制让 AI 自主操作文件系统完成代码开发任务。

---

## 项目概览（30 秒版本）

> 这是一个全栈项目，前端用 React 19 + ReactFlow 实现可视化流程编排，后端用 NestJS + TypeORM + MySQL 构建执行引擎。用户拖拽编排工作流节点，点击执行后系统按拓扑序自动运行，代码开发节点会启动 Agent Loop —— AI 自主循环调用 4 个文件工具，完成读、写、搜索、目录扫描，最终产出可 diff 审查的代码变更。整个过程通过 SSE 实时推送到前端展示。

## 核心模块与量化指标

### 1. Agent Loop 引擎（最核心亮点）

**架构设计：**
- 最大 **30 轮**循环迭代，每轮：调 AI → 解析 tool_calls → 执行工具 → 回填结果
- **4 个文件操作工具**：`list_directory`、`read_file`、`search_code`、`write_file`
- 单节点并发锁（`workflowId:nodeKey` 粒度），防止重复触发

**量化：**
- 平均 **8-15 轮**完成一个功能模块的代码生成
- 上下文压缩策略将 token 消耗降低约 **40%**（12 条消息阈值触发，保留最近 3 轮完整内容）
- `read_file` 源头截断：200 行软提示 + 500 行硬截断，单次工具返回 token 控制在 **2000 以内**

**面试话术：**
> Agent Loop 的设计参考了 Claude Code / Cursor 等工具的 agentic 模式。核心挑战是上下文管理 —— 跑 15 轮后 messages 可能膨胀到 5 万 token，我采用了"源头控制 + 滑动窗口"双重策略：源头上 read_file 做渐进式截断，窗口上保留最近 3 轮完整内容、早期轮次截短到 200 字符。选择不删消息而是截短，是因为 OpenAI API 要求 tool_call 和 tool_result 必须通过 tool_call_id 配对，删任何一条都会 400 报错。

---

### 2. 工作流执行引擎

**架构设计：**
- **简化 Kahn 算法**：计算入度 → 找起点 → 沿 nextMap 顺序执行
- **双路径分发**：根据 nodeType 分流到「文本生成」或「Agent Loop」
- **Human-in-the-Loop 审批**：执行暂停等待审批，approve 后 fire-and-forget 继续后续节点
- **文件快照/回滚机制**：执行前扫描目录生成 snapshot，执行后 diff 比对，reject 时精确回滚

**量化：**
- 支持 **7 种节点类型**：PRD 评审、需求评审、代码开发、自测、Code Review、提测、部署
- 快照覆盖 **整个工作目录**（排除 node_modules/.git/dist），reject 回滚精确到文件级别
- 审批响应 < **100ms**（fire-and-forget，后续节点异步执行）

**面试话术：**
> 执行引擎的核心设计是"快照 + diff + 审批"三件套。执行前我用递归扫描把整个工作目录序列化成 JSON 存数据库，执行后重新扫描做 diff，产出 added/modified/deleted 三种变更。前端用 react-diff-viewer 展示，审批通过则保留文件，拒绝则用快照精确回滚每个变更文件。这个设计避免了引入 memfs 虚拟文件系统的复杂度，trade-off 是磁盘 I/O 多一些。

---

### 3. SSE 流式推送

**架构设计：**
- 后端：`@Res()` 注入原始 Response，手动写 `text/event-stream` 头
- Agent 事件类型：`thinking`（轮次开始）→ `tool_done`（工具执行完成）→ `done`/`error`
- 前端：`fetch() + ReadableStream`（非 EventSource，因为需要 POST + Auth Header）
- 组件间通信：`CustomEvent('agent-log-event')` 跨 ReactFlow 渲染树传递

**量化：**
- 支持 **4 种事件类型**实时推送
- 前端 chunk 解析 + JSON.parse 实时渲染，延迟 < **50ms**

**面试话术：**
> 没用 EventSource 是因为它只支持 GET 且不能自定义 header。我用 fetch + ReadableStream 手动做 SSE 解析，通过 getReader() 逐 chunk 读取，按 `\n` 分割后 parse `data:` 行。比较有意思的设计是用 CustomEvent 做深层组件通信 —— ReactFlow 的自定义节点在渲染树很深的位置，和页面级组件没有 props 路径，用全局事件总线解耦了通信。

---

### 4. 三层 Prompt 系统

**设计：**
```
Layer 1: System（内置默认 prompt，按节点类型预设）
Layer 2: Project（项目级覆盖）
Layer 3: Node（节点级覆盖）
activeLayer: 1|2|3 → 取对应层，null 则降级到 system
```

**面试话术：**
> 三层设计解决了"开箱即用 vs 灵活定制"的矛盾。系统层为每种节点类型预设了高质量 prompt（比如代码开发节点会注入项目结构、tech stack 约束），用户可以在项目层或节点层覆盖。前端用 Monaco Editor 做 prompt 编辑，支持实时预览。

---

### 5. 前端可视化编排

**设计：**
- ReactFlow 画布 + 自定义节点/边组件
- HTML5 Drag API 实现节点面板拖拽
- Undo/Redo：history 数组 + 300ms 防抖快照
- 执行状态同步：3 秒轮询 + JSON 快照比较 + 10 次无变化自动停止

**面试话术：**
> 前端画布基于 ReactFlow，自定义了 StageNode 和 StageEdge 组件。比较有意思的是执行状态同步 —— 不用 WebSocket 而用轮询，因为当前规模下 3 秒轮询的实现成本远低于维护 WS 连接状态和重连逻辑。我做了优化：JSON 快照比对避免无效 re-render，连续 10 次无变化自动停止轮询。

| 措施 | 实现 |
|------|------|
| 路径穿越防护 | `path.resolve()` 归一化 + `startsWith()` 白名单检查 |

| 输入校验 | TypeORM parameterized query 防 SQL 注入 |

## 面试高频深问 & 应对

### Q: 为什么不用 LangChain？

> LangChain 抽象层太重，Tool-Calling 本质上就是一个循环 + JSON schema 匹配，OpenAI SDK 原生支持 tool_choice 和 function_calling，直接用 SDK 代码量反而更少，且可以精确控制每一轮的上下文处理逻辑。Agent Loop 核心代码不到 200 行。

### Q: 和 Cursor/Claude Code 的区别？

> 定位不同。Cursor 是单人 IDE 内的 AI 辅助，我这个是**团队协作的流程平台** —— 把软件开发拆成离散的节点，每个节点可以配不同的 prompt 和审批策略，适合标准化团队的开发流程。更像是 CI/CD pipeline 的 AI 版本。

### Q: 快照方案的性能瓶颈？

> 当前是递归扫描 + 全文存储，文件多了（>1000 文件或单文件 >1MB）会慢。改进方向：① 用 file hash 替代全文比对，只存变化的文件；② 用 git 做底层存储（每次执行自动 commit，reject = git revert）；③ 引入 memfs 做沙箱隔离。

### Q: 如果要支持并行分支怎么改？

> 当前 nextMap 是一对一映射（一个节点只记最后一条出边），改成一对多后用 Promise.all 并行执行分支节点，在汇聚节点（入度 > 1）做 barrier 等待所有上游完成。本质上从线性链升级为完整 DAG 执行。

### Q: 上下文压缩为什么不用 RAG？

> Agent Loop 的场景不适合 RAG：① 上下文是"对话历史"不是"知识库"，没有预构建的 embedding 索引；② 每轮工具调用的结果是即时生成的，无法提前向量化；③ 滑动窗口策略对"最近操作最重要"的 agent 模式更匹配。RAG 更适合静态知识检索场景。

## 项目亮点总结（面试结尾主动说）

1. **完整的 Agent 系统设计**：不是简单调 API，而是实现了 Tool-Calling 循环 + 上下文管理 + 并发控制的完整 Agent 引擎
2. **工程化的 LLM 输出处理**：两层正则容错、源头 token 控制、滑动窗口压缩，体现了对 AI 不确定性的工程化思维
3. **全链路 SSE 流式体验**：从 OpenAI SDK stream → NestJS 手写 SSE → fetch+ReadableStream 前端解析，端到端流式
4. **可审计的代码变更流程**：快照 → 执行 → diff → 审批/回滚，每一步可追溯
5. **全栈独立交付**：前后端 + 数据库 + AI 集成，monorepo 工程化管理，独立完成设计、开发、部署
