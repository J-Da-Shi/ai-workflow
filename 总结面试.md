# AI 编排项目
AI Dev Workflow — AI 驱动的软件开发工作流编排平台

项目简介

面向研发团队的 AI 工作流编排平台，将需求分析、PRD审核、代码开发等环节编排为自动化流水线，通过 AI Agent 自主编码 + RAG 知识库审核 + Git
自动化提交，减少重复性人工操作，提升研发流程标准化程度。

技术栈

React 19 + ReactFlow + Ant Design + react-diff-viewer NestJS + TypeORM + MySQL + Milvus（向量检索） OpenAI 兼容 API + text-embedding-v3 | SSE + simple-git + Zilliz Cloud

项目职责

1. 设计并实现 AI Agent 编码引擎：基于 Tool Use 协议实现 Agent Loop（文件读写/代码搜索/命令执行），
支持写完代码自动编译验证 + 失败自愈（最多 3 轮修复），单节点可自主完成 10+ 文件的创建与修改，替代人工逐文件编写，将重复性编码工作耗时从小时级降至分钟级
2. 落地 RAG 知识库驱动的 PRD 审核：完成文档处理全链路（解析 → 递归切片 → 批量 Embedding → Milvus
入库），审核时自动检索相关规范作为参考依据，使审核标准从主观判断转为可量化的结构化评分，覆盖文档完整性/技术可行性/边界条件等 4 个维度
3. 实现 Git 全流程自动化：封装 clone/branch/commit/push/PR 创建逻辑，适配 GitHub + GitLab 双平台，审批通过后自动推送代码并创建
PR，规避手动操作导致的分支命名混乱、忘记提交等问题，保障代码交付规范性
4. 设计上下文管理与安全防护机制：通过滑动窗口压缩 + 源头截断控制 Agent 上下文膨胀（支持 200 轮长任务不超窗口）；路径穿越校验 + 命令白名单确保 AI
操作可控，杜绝越权文件访问和危险命令执行
5. 实现 SSE 实时推送与 Diff 审批流程：Agent 执行过程实时可见（思考/工具调用/结果），完成后展示代码 Diff 供人工审查，驳回可一键还原，确保 AI 产出 100% 
经过人工确认后才进入代码库
6. 打通多节点自动串联执行：设计节点间结构化 JSON 协议（类型标识 + 内容 + 扩展字段），实现拓扑排序 + 审批控制 +
自动续跑，支持"需求→审核→开发"全链路无人值守执行，仅在关键节点暂停等待人工审批

  1. Agent Loop 相关
  
  可能的问法：
  - "你的 Agent Loop 和 Cursor 有什么区别？"
  - "为什么不直接调 Claude Code API？"
  - "你这个和 AI 代码编辑器有什么不同？"
  - "自己写 Agent Loop 的意义是什么？"
  
  回答：

  ▎ "架构原理完全一样 — while 循环 + tool_calls + 结果反馈。但定位不同：Cursor 是交互式的，人坐在电脑前实时对话；我的是流水线式的，用户配好 Prompt 就走了，Agent 
  ▎ 自动完成一整块任务，完成后等人审批。
  ▎
  ▎ 我的项目实际做了引擎可切换的设计 — 节点配置里可以选'自研 Agent Loop'或'Codex CLI'。自研用于简单任务（成本低、完全可控、可切任意模型），Codex 
  ▎ 用于复杂任务（内置工具更完整、效果更好）。如果 Codex CLI 环境不可用，自动降级到自研引擎，保证执行不中断。
  ▎
  ▎ 为什么不只接 Codex：第一，我需要平台层能力 — 多节点编排、审批流程、Git 自动化、知识库管理，这些外部引擎不提供。第二，自研引擎对每一步有完整控制 — 
  ▎ 白名单命令、路径安全校验、上下文压缩策略都是我决定的，不是黑盒。第三，模型可切 — 简单 CRUD 用便宜的 DeepSeek，复杂架构设计切到 GPT/Claude。
  ▎
  ▎ 自研过程中我也发现了一个核心效率差异：我的 write_file 要求 AI 输出完整文件内容（200 行文件 = 200 行 output token），而 Cursor 用 diff/patch 
  ▎ 模式只输出修改的几行。这是自研比 Cursor 慢的主要原因，后续优化方向是加 edit_file 工具支持增量编辑。"
    
  1.1 当Agent Loop 中，遇到简单的 CRUD 问题，我要用便宜的模型，复杂的再使用 GPT/Claude 这些，一般要怎么做？
    
  回答：

  ▎ "模型路由有三种粒度，从粗到细：
  ▎
  ▎ 第一种：节点级路由（我当前的做法）
  ▎
  ▎ 在节点配置中让用户选模型。简单节点（需求分析、普通 AI 对话）配便宜模型（GPT-mini），复杂节点（代码开发）配强模型（GPT-4/Claude）。判断权交给用户，零额外调用成本。
  ▎
  ▎ 第二种：任务级路由（自动判断）
  ▎
  ▎ 不是用 AI 判断，而是用规则判断。比如：
  ▎ - 上游输入 < 500 字 + 节点 Prompt < 200 字 → 判定为简单任务 → 用便宜模型
  ▎ - 上游输入包含"架构设计"、"多文件"、"重构"等关键词 → 判定为复杂任务 → 用强模型
  ▎ - 代码开发节点如果 Git 仓库文件数 > 50 → 复杂 → 强模型
  ▎
  ▎ 这种用规则引擎判断，不需要调 AI，零成本零延迟。
  ▎
  ▎ 第三种：Loop 内动态切换（最精细但最复杂）
  ▎
  ▎ Agent Loop 的前几轮（list_directory、read_file 这些探索性操作）用便宜模型，到了要 write_file 生成代码时切强模型。实现方式是在 Loop 循环中根据当前轮次或上一轮 AI 返回的 
  ▎ tool_calls 类型决定下一轮用哪个模型。
  ▎
  ▎ 但这有风险：不同模型对 messages 历史的理解可能不同，中间切模型可能导致上下文'断层'。更稳妥的做法是整个 Loop 用同一个模型，只在节点级做路由。
  ▎
  ▎ 我当前选的是第一种 — 用户在节点配置中选模型/引擎，简单可控。如果要做自动路由，我倾向第二种（规则判断），不倾向让 AI 自己判断自己。"

  1.2 "为什么不在 Loop 中间动态切？"

    回答：

  ▎ "两个原因：第一，messages 兼容性 — 不同模型对 tool_calls 格式的理解可能有差异，中途换模型可能导致解析错误。第二，上下文连贯性 — 模型 A 在第 5 轮基于前 4 
  ▎ 轮的理解做了决策，换成模型 B 后它没有前 4 轮的'思维连贯性'，即使 messages 一样，推理路径也不同。
  ▎
  ▎ 如果一定要 Loop 内切换，比较安全的做法是：用便宜模型跑完整个探索阶段（前 N 轮都是 read/list/search），生成一份'分析摘要'，然后开一个新的 Loop（新的 
  ▎ messages），把摘要作为输入传给强模型做生成阶段。本质上是两段式执行，而不是一个 Loop 内动态切换。"

  2. RAG 检索质量
  
  可能的问法：
  - "RAG 检索质量怎么保证？"
  - "你怎么知道检索到的内容是对的？"
  - "检索效果有量化指标吗？"
  - "向量检索的精度怎么评估？"
  
  回答：

  ▎ "当前靠三层过滤：第一层是 Milvus 的向量相似度 top-k；第二层是 score 阈值 0.3（COSINE 度量下低于 0.3 的基本无关）；第三层是 Prompt 中约束 AI 
  ▎ '仅基于参考资料回答，材料中没有的不要编造'。
  ▎
  ▎ 但坦白说，我没有正式的量化评估 — 没有标注 query-relevant_chunk 的测试集，不知道 Recall@5 具体是多少。我的验证方式是：手动输入几个测试 query 看返回的 chunk 
  ▎ 是否相关，人工判断命中率。这对开发阶段够用，但不够严谨。
  ▎
  ▎ 如果要生产化，需要做三件事：第一，准备 20-50 个标注好的测试 query + 期望命中的 chunk，算 Recall@k 和 MRR；第二，加 Rerank 模型（如 bge-reranker），对 top-k 
  ▎ 结果做二次排序，代价是每次检索多 100-200ms 但精度显著提升；第三，在审核报告中展示引用来源，让用户能判断 AI 是不是基于相关资料回答的。
  ▎
  ▎ 还有一个实际踩的坑：阿里云 Embedding API 每批限制 10 条输入，我一开始按 OpenAI 文档设了 
  ▎ 20，整个批次直接报错。这种限制在文档里不显眼，是看报错信息才定位到的。修复方法很简单 — batchSize 从 20 改 10，但排查花了不少时间。"

  3. 安全性
  
  可能的问法：
  - "AI 会不会搞破坏？"
  - "怎么防止 Agent 执行危险操作？"
  - "路径穿越怎么防的？"
  - "为什么用白名单不用黑名单？"
  
  回答：

  ▎ "三个层面的防护：
  ▎
  ▎ 文件层：所有路径操作先过 safePath 函数。具体做法是 path.resolve(workDir, filePath) 把 .. 全解析掉得到真实绝对路径，然后检查这个路径是否以 workDir 开头。如果 AI 传入 
  ▎ ../../etc/passwd，resolve 后变成 /etc/passwd，不以 workDir 开头 → 直接拒绝。这个方案简单但有效，覆盖了所有路径穿越变体。
  ▎
  ▎ 命令层：run_command 用白名单。只允许 npm run build、npm test、npx tsc --noEmit 等预定义命令前缀，其他一律返回'不允许执行'。为什么白名单不用黑名单 — 黑名单永远有遗漏，rm 
  ▎ 可以写成 \rm、$(rm ...)、反引号嵌套，AI 的组合能力比人想象的强。白名单是唯一能保证 100% 安全的方案。另外命令有 60 秒超时，防止 AI 执行一个 while true 卡住进程。
  ▎
  ▎ Git 层：Token 不暴露给 Agent — Git 操作（clone/commit/push）是执行引擎的前后置钩子，不是 AI 能调用的工具。AI 在 Agent Loop 中看不到 token，也无法执行 git push --force 或 
  ▎ git branch -D。
  ▎
  ▎ 还有一个我设计时考虑但没做的：文件大小限制。如果 AI 写一个 10MB 的文件，当前 write_file 不会拒绝。生产环境应该加单文件上限（如 1MB）和单次执行总写入量限制。"

  4. Diff 审批
  
  可能的问法：
  - "Diff 审批怎么实现的？"
  - "为什么不直接用 Git diff？"
  - "全目录扫描性能怎么样？"
  - "驳回的时候怎么还原文件？"
  
  回答：

  ▎ "实现方式是执行前后对比：
  ▎
  ▎ 执行前，scanDirectory 递归扫描工作目录，把每个文件的路径和内容存成一个 Map（{ 'src/app.ts': '文件内容...' }），保存到 MySQL 的 snapshot 字段（JSON 格式）。扫描时排除 
  ▎ node_modules、.git、dist 这三个目录（占文件量的 99%），单文件超过 512KB 不读内容。
  ▎
  ▎ 执行后，再次扫描得到 current Map，和 snapshot 对比：current 有而 snapshot 没有 = added；两边都有但内容不同 = modified；snapshot 有而 current 没有 = deleted。生成一个 
  ▎ changes 数组存到 execution 记录。
  ▎
  ▎ 前端用 react-diff-viewer 渲染 before/after 对比视图。用户审批驳回时，后端根据 changes 逐个还原：added 的文件直接 fs.rmSync 删除，modified 的用 snapshot 
  ▎ 中的旧内容覆盖写回，deleted 的从 snapshot 恢复。
  ▎
  ▎ 为什么不用 Git diff：不是所有节点都配了 Git。有些用户只想让 AI 写几个文件看看效果，不想配仓库。我的改进方案是两条路并存 — 如果配了 Git 就用 git diff 
  ▎ HEAD（更快更标准），没配 Git 才走快照方案。当前实现只有快照方案。
  ▎
  ▎ 性能方面：排除了三大目录后，一个正常项目几十个源文件，扫描 < 100ms。几千个文件的大项目确实会慢，但这种项目一定会配 Git，走 Git diff 就不存在性能问题。"

  5. LangChain
  
  可能的问法：
  - "为什么不用 LangChain？"
  - "你了解 LangChain 吗？和你的实现有什么区别？"
  - "RAG 为什么不用现成框架？"
  
  回答：

  ▎ "评估过 LangChain，最终决定不用，三个原因：
  ▎
  ▎ 第一，链路太短不需要编排框架。我的 RAG 全链路是 解析 → 切片 → Embedding → Milvus 入库 → 搜索，每一步对应一个具体的库调用（pdf-parse / OpenAI SDK / Milvus SDK），加起来 100
  ▎  行代码。LangChain 在每一步上包了 Document → BaseLoader → TextSplitter → VectorStore → Chain 共 4-5 层抽象，出了问题要翻好几层源码定位。
  ▎
  ▎ 第二，需要精细控制。Embedding 批量大小我要设成 10（阿里云限制），重试逻辑我要指数退避（1s → 2s → 4s），切片策略我要 Markdown 按标题优先分割。这些在 LangChain 中要 override
  ▎  默认行为或写自定义 Splitter，不如直接自己写 20 行来得清晰可控。
  ▎
  ▎ 第三，依赖膨胀。LangChain 的 node_modules 有几十 MB，拖了 Agent/Memory/Callback 等我完全用不到的模块。我只装了 pdf-parse + milvus-sdk + openai 三个包。
  ▎
  ▎ 什么时候该用 LangChain：如果需要快速实验多种检索策略（BM25 / 向量 / 混合）、对比不同 LLM 的效果、或者做复杂的多轮对话 + 检索 + 工具调用的编排，LangChain 
  ▎ 的统一抽象确实省事。但我的场景确定性强、链路固定，自己写更合适。"

  6. 容错机制

  可能的问法：
  - "节点执行失败怎么办？"
  - "有容错和重试机制吗？"
  - "工作流中间断了能恢复吗？"
  
  回答：

  ▎ "容错分三层：
  ▎
  ▎ 工具执行层：Agent Loop 中每个工具调用都 try/catch 包裹，异常不会中断循环。如果 read_file 读了一个不存在的文件，返回'文件不存在'的文本给 AI，AI 看到后会调整策略（比如先 
  ▎ list_directory 确认文件名），而不是整个 Loop 崩掉。
  ▎
  ▎ 节点执行层：每个节点的 executeNode 整体在 try/catch 中，失败时 execution.status = 'failed'、error 
  ▎ 字段记录具体错误。工作流停在失败节点，不会继续执行下游。用户能看到报错，修改 Prompt 后重新触发。
  ▎
  ▎ 并发保护层：我遇到过一个真实问题 — 用户连点两下执行按钮，导致两个 Agent Loop 同时跑在同一个 workDir 里互相覆盖文件。解决方式是加并发锁，用 Map 存 workflowId:nodeKey 作为锁
  ▎  key，第二次触发时直接抛错'该节点正在执行中'。
  ▎
  ▎ 几个已知不足：第一，LLM API 超时没有自动重试（生产级应该对 5xx 重试 2-3 次）；第二，没有外部超时熔断（如果 AI 进入低效循环，只能等 200 
  ▎ 轮上限自动停止）；第三，断点续跑不完整 — 工作流跑到第 3 个节点失败后不会自动重跑前面已通过的节点（通过检查 status = 'approved' 跳过），但中间失败的节点需要手动重新触发。"

  7. 并发与性能
  
  可能的问法：
  - "系统能支持多少并发？"
  - "瓶颈在哪？"
  - "Node.js 单线程不是问题吗？"
  - "有没有遇到过性能问题？"
  
  回答：

  ▎ "瓶颈不在服务端计算，在 AI API 调用。
  ▎
  ▎ Node.js 虽然单线程，但 I/O 全是 async/await。10 个用户同时触发 Agent Loop，10 个 fetch 请求同时 await AI API 响应，互不阻塞。CPU 几乎不消耗（只做 JSON 解析和字符串拼接）。
  ▎
  ▎ 实际瓶颈是 AI API 的 RPM 限制。每个 Agent Loop 每轮发一次请求，10 个并发 × 每秒 1 次 = 600 RPM，容易触发限流。解决方案是加请求队列做限流（比如 p-queue 控制并发数）。
  ▎
  ▎ 我遇到的真实性能问题是 Agent Loop 执行太慢 — 一个 500 行的代码生成任务跑了 10 分钟。分析后发现原因是：每轮 AI 响应 3-5 秒 × 15-20 轮 = 分钟级。加上 write_file 要求 AI 
  ▎ 输出完整文件内容（大量 output token），比 diff 模式慢很多。优化方向是加 edit_file 工具（只传修改片段）和换更快的模型。
  ▎
  ▎ 如果要生产部署：1. PM2 cluster 模式跑多进程分担连接；2. AI API 调用加限流队列；3. 长任务（Agent Loop）改为后台任务，HTTP 立即返回，用 SSE 推进度。"

  8. 节点间数据一致性
  
  可能的问法：
  - "节点之间的数据传递怎么做的？"
  - "上游重新执行了下游会不会用到旧数据？"
  - "数据一致性怎么保证？"
  
  回答：

  ▎ "节点间传递靠结构化 JSON。每个节点执行完把输出存到 execution.output — 是一个 JSON 字符串，格式如 { type: 'code_dev', text: 'AI总结', changedFiles: [...] }。下游执行时通过 
  ▎ getNodeInput 读取上游的 output，JSON.parse 提取 .text 字段（自然语言内容）传给 AI。
  ▎
  ▎ 数据一致性是快照式的 — 下游读到的永远是它执行那一刻上游的最新 
  ▎ output。如果用户后来重跑了上游（生成了新结果），下游不会自动感知。这是有意的设计：自动级联重跑可能覆盖下游已经审批通过的代码，风险太大。应该由用户决定是否重跑下游。
  ▎
  ▎ 向后兼容也做了：getNodeInput 里 try JSON.parse，如果上游是旧数据（纯文本不是 JSON），catch 后直接返回原文。这样新旧格式的节点可以混用，不需要全部重跑。
  ▎
  ▎ 改进方向是加版本标记 — 下游执行时记录'我引用的是上游第 N 次执行的结果'，如果上游更新了，前端标黄提示'上游已更新，当前结果可能过期'。"

  9. 切片策略
  
  可能的问法：
  - "切片大小怎么选的？"
  - "为什么要 overlap？"
  - "有做过 A/B 测试吗？"
  - "Markdown 文档怎么切？"
  
  回答：

  ▎ "切片大小 512 token（约 1500 中文字符），重叠 200 字符。这个数字参考 OpenAI cookbook 和 LlamaIndex 的测试结论 — 他们在 QA 任务上对比了 128/256/512/1024/2048 五个粒度，512 
  ▎ 的 Recall@5 和回答质量综合最优。
  ▎
  ▎ 原理很直觉：切太小（128）每个 chunk 信息量不够，检索到了也拼不出完整答案；切太大（2048）一个向量包含多个话题，语义模糊，检索精度下降。
  ▎
  ▎ 重叠 200 字符是为了防止关键句子在边界被截断。比如'接口返回 JSON，状态码 200 表示成功'如果恰好在逗号处被切成两个 chunk，搜索'接口返回的状态码'时两个 chunk 
  ▎ 都不完整。有重叠后前一个 chunk 的尾部和后一个 chunk 的头部有共同内容，命中率更高。
  ▎
  ▎ 切片算法是递归字符分割：先按段落（\n\n）分，如果段落太长再按换行（\n）分，还是太长按字数硬切。Markdown 文档天然按标题优先 — \n\n 在标题前后必然存在，所以自动按章节切割。
  ▎
  ▎ 我没做 A/B 对比实验。如果要做：准备 50 个 query + 标注答案所在段落，用不同 chunkSize 分别入库跑 Recall@5 对比。这是后续优化方向，当前 512 的效果从人工检查来看是 OK 的。"

  ▎
  ▎ 我做了两条路互为备份：实时通道是 SSE（onEvent 推送 thinking/tool_done/done 事件），持久化通道是 agent_logs 表（每次工具调用都入库）。用户刷新页面后，nodeRun 组件从 GET API
  ▎  加载历史日志，能看到完整的执行过程。

  ▎  tab 中看到进度。
  ▎
  ▎ 为什么不用 WebSocket：这是单向推送场景（服务端 → 客户端），不需要客户端通过同一连接回传数据。SSE 基于 HTTP，不需要额外的连接升级/心跳/重连逻辑，实现成本低得多。另外 SSE 
  ▎ 天然兼容 HTTP 中间件（鉴权、代理），WebSocket 需要单独处理。
  ▎
  ▎ 为什么不用浏览器原生 EventSource：因为 EventSource 只支持 GET 且不能带自定义 Header。我需要 POST（触发执行）+ Authorization Header（JWT 鉴权），所以用 fetch + getReader 
  ▎ 手动读流。"

  12. JSON 字符串 vs 结构化字段
  
  可能的问法：
  - "为什么不用 MySQL JSON 类型？"
  - "output 存 JSON 字符串不会有问题吗？"
  - "这个设计有什么缺陷？"
  
  回答：

  ▎ "两个考虑：
  ▎
  ▎ 向后兼容：output 字段最初是 TEXT 存纯文本（早期节点的执行结果）。如果改成 JSON 类型需要 ALTER TABLE + 旧数据全部不兼容。用 JSON.stringify 存在 TEXT 里，新数据是 JSON 
  ▎ 格式，旧数据还是纯文本 — getNodeInput 里 try JSON.parse catch 后 fallback 返回原文，零 migration 成本。

  ▎ 再过滤。如果后续有统计分析需求，需要加冗余字段（如 execution 表加一个 verdict 列）或用 MySQL 8 的 JSON 函数（JSON_EXTRACT）。当前规模不需要，所以选了灵活方案。"

  18. 项目时间规划
  
  可能的问法：
  - "花了多长时间？"
  - "怎么规划的？"
  - "一个人做了这么多功能？"
  
  回答：

  ▎ "大约 3 周业余时间。规划原则是先跑通最小闭环，再叠加能力。
  ▎
  ▎ 第一周做了 Agent Loop + 文件工具 + SSE 推送。为什么 SSE 排第一周 — 没有实时反馈用户根本不知道 Agent 在做什么，执行 3 分钟没任何进度，体验完全是黑盒。SSE

  ▎ 审批是'质量保障'，保障比交付更优先。

  ▎ 第三周做了 RAG（Milvus + Embedding + 知识库 + PRD审核）+ 前端重构。RAG 放最后是因为即使没有知识库，PRD审核节点也能走普通 AI 调用先跑起来，不影响整体闭环。

  # RAG 相关

  1. "你的 RAG 整体架构是什么？数据怎么流转的？"
  
  ▎ "分两条链路：
  ▎
  ▎ 入库链路：用户上传文档（PDF/Word/MD）→ DocumentProcessorService 解析提取纯文本 → splitText 递归切片（1500 字符/片，200 重叠）→ EmbeddingService 批量转向量（阿里云 
  ▎ text-embedding-v3，1024 维）→ 双写存储（Milvus 存向量用于搜索，MySQL 存原文用于管理）。
  ▎
  ▎ 检索链路：PRD 审核节点执行 → 把 PRD 文本调 embedText 转向量 → MilvusService.search（filter 限定知识库范围，COSINE 相似度，top-8）→ 过滤 score < 0.3 的噪声 → 拼接到 System 
  ▎ Prompt 的'参考资料'段落 → AI 基于增强后的 Prompt 输出审核报告。
  ▎
  ▎ MySQL 和 Milvus 通过 chunk ID 关联（同一个 UUID），删除文档时两边同步清理。"

  2. "为什么选 Milvus？和 Chroma、pgvector 比有什么优势？"
  
  ▎ "三个原因：
  ▎
  ▎ 第一，多租户隔离。我的场景有多个知识库（PRD审核标准、代码规范等），检索时需要 filter 限定范围。Milvus 原生支持标量过滤（knowledge_base_id in 
  ▎ [...]），在向量搜索之前先缩小范围，性能好。Chroma 的过滤能力弱，pgvector 要走 SQL WHERE + 向量排序的组合查询，复杂且慢。
  ▎
  ▎ 第二，生产级特性。Milvus 支持分区、持久化、水平扩展、索引多样性（IVF/HNSW/PQ）。Chroma 定位是轻量嵌入式（适合原型），不适合生产部署。
  ▎
  ▎ 第三，我用的是 Zilliz Cloud（Milvus 的托管版），Serverless 免运维，有免费额度。不需要自己 Docker 部署 Milvus + etcd + MinIO 那一套。
  ▎
  ▎ pgvector 的优势是'不用引入新组件'（直接在 PostgreSQL 里加扩展），但我用的是 MySQL 不支持 pgvector。"

  3. "切片策略怎么设计的？为什么这样切？"
  
  ▎ "递归字符分割，三层优先级：
  ▎
  ▎ 第一层按段落（\n\n）分 — 段落是最自然的语义边界，同一段通常讨论同一个话题。第二层对超长段落按换行（\n）再切。第三层还是太长就按字数硬切。
  ▎
  ▎ chunk 大小 1500 字符（约 512 token），重叠 200 字符。
  ▎
  ▎ 为什么 512 token：参考 OpenAI cookbook 的测试结论，在 QA 任务上 512 的 Recall@5 综合最优。太小（128）信息量不够，太大（2048）语义太模糊。
  ▎
  ▎ 为什么要重叠：防止关键句子在边界被截断。比如'接口返回 JSON，状态码 200 表示成功'如果在逗号处被切断，两个 chunk 都不完整。有 200 字符重叠后，边界附近的内容在相邻 chunk 
  ▎ 中都存在，检索命中率更高。
  ▎
  ▎ Markdown 文档天然适配这个策略 — 标题前后必有 \n\n，所以会自动按章节切割。"

  4. "Embedding 模型怎么选的？为什么用阿里云不用 OpenAI？"
  
  ▎ "实际原因是成本和可用性：我的 Chat 模型用中转站，但中转站没有 Embedding 模型。阿里云百炼的 text-embedding-v3 免费额度够用，且 API 兼容 OpenAI SDK 格式（直接用 
  ▎ openai.embeddings.create），接入零成本。
  ▎
  ▎ 维度是 1024。如果换 OpenAI 的 text-embedding-3-small（1536 维）或 text-embedding-3-large（3072 维），精度更高但需要重建 Milvus Collection + 重新 embedding 所有文档。
  ▎
  ▎ 代码设计上做了解耦：EmbeddingService 有独立的 EMBEDDING_API_KEY 和 EMBEDDING_BASE_URL 环境变量，和 Chat 模型完全隔离。换 Embedding 提供商只改 .env + 改 Collection 
  ▎ 维度就行。"

  5. "检索时 score 阈值 0.3 怎么确定的？为什么不是 0.7？"
  
  ▎ "COSINE 相似度的范围是 0-1，但文本检索的实际命中分数分布通常在 0.3-0.8 之间。完全相同的文本是 1.0，完全无关的是 0.0-0.2。
  ▎
  ▎ 0.3 是比较宽松的阈值 — 宁可多给一些'可能相关'的结果让 AI 自己判断，而不是严格过滤掉潜在有用的内容。如果设 0.7 太严格，知识库里稍微换个表述方式（如'接口设计' vs 'API 
  ▎ 规范'）就可能低于 0.7 被丢弃。
  ▎
  ▎ 这个阈值是我手动测试几个 query 后拍的，看返回结果中 score 的分布。真正严谨的做法是准备标注数据集，画 precision-recall 曲线找最优 threshold。这是后续优化方向。"

  ---
  6. "如果知识库里没有相关内容，AI 会怎么处理？会不会瞎编？"
  
  ▎ "两层防护：
  ▎
  ▎ 第一层，score 阈值过滤。如果知识库里确实没有相关内容，Milvus 返回的 top-k 结果 score 都很低（< 0.3），全部被过滤掉，retrievedContext 为空。此时 Prompt 
  ▎ 中没有'参考资料'段落，AI 只基于自己的知识审核 — 效果和没有 RAG 时一样。
  ▎
  ▎ 第二层，Prompt 约束。buildReviewPrompt 中明确告诉 AI'仅基于提供的参考资料回答，如果材料中没有相关信息请明确说明'。这不能 100% 防止幻觉，但能显著降低 AI 编造内容的概率。
  ▎
  ▎ 终极解决方案是：在审核报告中附上引用来源（'基于参考资料 1，相似度 72%'），让用户能判断 AI 的结论是否有据可依。我的 retrievedContext 拼接时已经标注了每条的相似度百分比。"

  7. "向量数据库的索引怎么选的？IVF_FLAT 参数怎么调？"
  
  ▎ "选 IVF_FLAT 是因为当前数据量小（< 1 万条向量）。IVF 先把向量聚类成 N 个桶（nlist=128），搜索时只扫描最近的几个桶（nprobe=16），不用暴力扫全部。
  ▎
  ▎ nlist=128：数据量 1 万级时 128 个聚类够用，每个桶平均约 80 条向量。如果涨到百万级，应该加到 1024。
  ▎
  ▎ nprobe=16：搜索时扫描 128 个桶中最近的 16 个（12.5%）。值越大越精确但越慢。16 是精度和速度的平衡点 — 搜索延迟 < 50ms。
  ▎
  ▎ 度量选 COSINE 不选 L2：文本 embedding 的'方向'比'长度'有意义。两段语义相近的文本向量方向相似，但长度可能因文本长短而不同。COSINE 归一化了长度只比方向。
  ▎
  ▎ 如果数据量涨到百万级会换 HNSW（精度更高、搜索更快，但内存占用翻倍）。"

  ---
  8. "文档上传到入库的过程中，如果某一步失败了怎么办？"
  
  ▎ "每个文档有独立的 status 状态机：pending → processing → completed / failed。
  ▎
  ▎ 处理管线整体在 try/catch 中。如果任何一步失败（PDF 解析报错、Embedding API 超时、Milvus 写入失败），catch 中把 document.status 设为 'failed'，error 
  ▎ 字段记录具体错误信息。用户在前端能看到哪个文档失败了、失败原因是什么。
  ▎
  ▎ Embedding 调用做了重试 — 每批最多重试 3 次，指数退避（1s → 2s → 4s）。网络抖动导致的偶发失败不会直接标记为 failed，重试后成功就正常继续。
  ▎
  ▎ 部分成功的情况：比如 50 个 chunk 中前 30 个入库成功，第 31 个 Embedding 失败。当前整体标记为 failed，已入库的 30 个不会回滚（Milvus 中有数据但 MySQL 标记 
  ▎ failed）。严格来说应该做事务回滚，但当前规模影响不大 — 用户重新上传会生成新的 chunk ID，不会和旧数据冲突。"

  9. "你的 RAG 和直接把文档全文塞给 AI 有什么区别？为什么不直接塞全文？"
  
  ▎ "两个问题：
  ▎
  ▎ 第一，上下文窗口限制。一份 PRD 规范文档可能有 5000-10000 字，加上 PRD 本身也有几千字，直接塞全文很容易超出模型的上下文窗口（GPT-4 turbo 是 
  ▎ 128K，但实际推理质量在超长上下文下会显著下降）。
  ▎
  ▎ 第二，噪声干扰。一份 10 页的规范文档中，和当前 PRD 审核相关的可能只有 2-3 段。把无关的 7-8 页也塞进去会分散 AI 的注意力（Needle in a Haystack 问题），审核质量反而下降。
  ▎
  ▎ RAG 的价值就是'精准投喂'— 只把最相关的 8 个片段给 AI，而不是整本书。token 消耗少，AI 注意力集中，回答质量更高。
  ▎
  ▎ 当然如果知识库只有 1-2 页内容，直接塞全文更简单有效。RAG 的价值在知识库内容多（几十上百份文档）时才真正体现。"

  10. "如果要提升检索质量，你会怎么做？"
  
  ▎ "三个方向：
  ▎
  ▎ 第一，加 Rerank。当前 top-k 完全靠 embedding 向量相似度排序，但 embedding 是通用的'语义相似'，不一定等于'对审核有用'。加一个 Cross-Encoder reranker（如 
  ▎ bge-reranker-v2），对 top-20 初筛结果做精排，重新按'和审核任务的相关性'排序，取 top-8。代价是每次检索多 100-200ms，精度提升显著。
  ▎
  ▎ 第二，混合检索。当前只用稠密向量（embedding），漏掉了关键词精确匹配的场景。比如搜索'JWT 认证'，embedding 可能返回'Token 鉴权'（语义相近但没提到 JWT 这个词）。加 BM25 
  ▎ 稀疏检索做互补，用 RRF（Reciprocal Rank Fusion）融合两路排序结果。Milvus 2.4+ 原生支持 sparse vector。
  ▎
  ▎ 第三，Query 改写。用户的输入可能很短或很模糊（如'安全'），embedding 效果差。先用 LLM 把 query 
  ▎ 改写为更具体的检索语句（如'接口安全认证方案、数据加密、权限校验'），再去检索。相当于用 AI 理解用户意图后再搜索。"

  11. "双写（Milvus + MySQL）一致性怎么保证？"
  
  ▎ "当前没有严格的分布式事务，靠写入顺序保证最终一致性：
  ▎
  ▎ 入库时先写 Milvus 再写 MySQL。如果 Milvus 成功但 MySQL 失败（极小概率），Milvus 中会有孤立数据（搜索能搜到但管理页面看不到）。影响不大因为数据还是可用的。
  ▎
  ▎ 删除时先查 MySQL 拿到 chunk ID 列表，再调 Milvus deleteByFilter 删向量，最后删 MySQL（CASCADE 自动删 chunks）。如果 Milvus 删除失败，MySQL 仍然删了 — 这种情况 Milvus 
  ▎ 中有垃圾数据。目前不处理，因为这些数据虽然存在但不会被检索到（filter 靠 knowledge_base_id，知识库已删就不会被选中）。
  ▎
  ▎ 严格来说应该用 Saga 模式（先删 MySQL 标记为 deleting，再删 Milvus，最后确认删除或回滚）。但当前规模不需要这个复杂度。"

  12. "你怎么处理文档更新？同一个文档重新上传怎么办？"
  
  ▎ "当前没有'更新'概念 — 重新上传同一个文件会生成新的 document 记录和新的 chunk（新 UUID）。旧的需要手动删除。
  ▎
  ▎ 如果要做'更新'：先删旧的（MySQL chunks + Milvus 向量）→ 重新解析切片 → 重新 embedding 入库。本质上是 delete + insert，不是 
  ▎ update。因为文档内容变了，切片结果完全不同，没法对应旧 chunk 做局部更新。
  ▎
  ▎ 优化方向是加 content hash — 上传时计算文件 hash，如果和已有文档的 hash 相同说明内容没变，跳过处理。这避免了重复上传导致的重复数据。"

  ▎
  ▎ 实际效果：跑 20 轮后 messages 条数还是 20+，但总 token 数被控制住了 — 早期轮次的大段文件内容被截断为摘要。"

  MCP 接入方案（面试版）
  
  一句话

  ▎ 让 Agent 的工具列表从"固定 5 个"变成"动态加载"，用户配一个 MCP Server 地址，平台自动获取该 Server 暴露的工具，合并到 AI 的 tools 参数中。

  架构

  用户配置 MCP Server 地址（如 Jira MCP、数据库 MCP）
    → 平台启动时连接 Server，获取工具列表（name + description + parameters）
    → 合并到 Agent Loop 的 tools 数组中（内置 5 个 + MCP N 个）
    → AI 决定调用某个 MCP 工具
    → 平台判断：是内置工具 → 本地执行；是 MCP 工具 → 转发给对应 Server 执行
    → 拿到结果返回给 AI

  关键设计点

  - 协议标准化：MCP Server 暴露 GET /tools（返回工具列表）和 POST /call-tool（执行工具），平台只需实现客户端
  - 动态加载：工具列表不再是代码中的常量，而是每次执行前从配置的 MCP Server 动态拉取
  - 分流执行：executeTool 中判断工具来源 — 内置的本地执行，MCP 的走网络转发
  - 安全边界：MCP 工具的权限由 MCP Server 自己控制，平台只做转发。但可以在平台层加白名单（只允许接入审批过的 MCP Server）
  - 配置层级：MCP Server 可以配在项目级（整个项目共享）或节点级（某个节点专用）

  面试怎么讲

  ▎ "我的 Agent Loop 工具列表当前是静态的 5 个。如果要支持 MCP，核心改动是把工具列表从静态常量改为动态加载 — 节点执行前从配置的 MCP Server 拉取工具定义，合并到 tools 
  ▎ 参数中传给 AI。AI 调用 MCP 工具时，我在 executeTool 层做分流：内置工具本地执行，MCP 工具转发给对应 Server。对 AI 来说是透明的 — 它不知道工具是内置的还是 MCP 的，只看到 
  ▎ tools 数组里多了几个可用工具。"

  Skill 体系方案（面试版）
  
  一句话

  ▎ Skill 是"可复用的节点配置模板"，用户选一个 Skill 自动填充 Prompt + 工具配置 + 输出格式，不用从零写。

  架构

  Skill 库（系统内置 + 团队自定义 + 社区共享）
    → 用户创建节点时选一个 Skill
    → 自动填充：Prompt 模板 + 推荐模型 + 工具白名单 + 输出格式要求
    → 用户可微调（改 Prompt、换模型）
    → 执行时按 Skill 定义的策略运行

  Skill 的数据结构

  Skill = {
    name: "React 组件生成",
    category: "前端开发",
    promptTemplate: "你是前端工程师，请创建一个 {{componentName}} 组件...",
    variables: ["componentName"],           // 用户需要填的变量
    recommendedModel: "gpt-5.4-mini",       // 推荐模型
    recommendedEngine: "agent",             // 推荐引擎
    outputFormat: "创建文件列表 + 每个文件的作用", // 期望输出格式
    knowledgeBaseIds: [...],                // 关联的知识库（可选）
  }

  和现有架构的关系

  - Skill 本质上就是对 nodeConfig 的预填充 — 选了 Skill 后把 promptTemplate 填到 Layer 1（系统默认），用户在 Layer 3 可以覆盖
  - 不改执行引擎 — Skill 只影响配置阶段，执行时还是走正常的 Agent Loop / Codex
  - 变量替换 — Prompt 中的 {{componentName}} 在执行前由用户输入或上游输出自动替换

  面试怎么讲

  ▎ "Skill 是我的 Prompt 三层配置的自然延伸。当前 Layer 1 系统默认 Prompt 是写死的通用规则，如果把它改成可选的 Skill 模板 — 用户创建节点时从 Skill 库选一个（如'React 
  ▎ 组件生成'、'API 接口开发'、'代码 Review'），自动填充 Prompt + 推荐模型 + 关联知识库。用户在 Layer 3 还可以覆盖微调。本质上 Skill 
  ▎ 就是'节点配置的预设模板'，降低使用门槛，让不会写 Prompt 的人也能用好平台。"

  ▎ "workflow_nodes 是配置数据（Prompt、模型、Git 配置），是静态的，用户编辑时写入。node_executions 是运行时数据（状态、输出、快照），每次执行都产生一条新记录。
  ▎
  ▎ 分开的好处：一个节点可以执行多次（修改 Prompt 后重跑），每次执行是独立的记录。配置改了不影响历史执行记录的回溯。如果合一张表，每次执行都要 update 同一行，丢失历史。"

  "canvasData 为什么存 JSON 不拆表？"


  ▎ "canvasData 包含画布上所有节点的位置坐标和连线信息，结构是 { nodes: [{id, position, data}], edges: [{source, target}] }。这是 ReactFlow
  ▎ 的原始数据格式，前端直接存直接取，不需要服务端按字段查询。拆成关系表（canvas_nodes + canvas_edges）增加复杂度但没有收益 — 从来不会 WHERE 某个节点的 x 坐标。"

  "node_executions 的 output 为什么是 TEXT 存 JSON 字符串？"

  ▎ "向后兼容。最初设计时 output 存纯文本（AI 的回复原文）。后来加了结构化协议（{ type: 'code_dev', text, changedFiles }），用 JSON.stringify 存在 TEXT 字段里 —
  ▎ 旧数据是纯文本也不会坏，getNodeInput 里 try JSON.parse catch 后 fallback 返回原文。如果一开始用 JSON 类型，旧的纯文本记录全部不兼容。"

  "knowledge_chunk 为什么 MySQL 和 Milvus 都存？"

  ▎ "各管各的：Milvus 存向量用于相似度搜索（几毫秒返回 top-k），MySQL 存原文用于管理（文档列表展示、删除级联、状态查询）。两边用同一个 UUID 关联。
  ▎
  ▎ 如果只存 Milvus：管理页面要展示切片内容就得查 Milvus（它不擅长精确查询和分页）。如果只存 MySQL：搜索时要把所有 chunk 的文本取出来逐一 embedding +
  ▎ 比较（不现实）。各存各的是标准的 RAG 双写模式。"

  "agent_logs 和 node_executions 的关系？"

  ▎ "一对多。一个 node_execution（一次执行）对应多条 agent_logs（多次工具调用）。通过 workflowId + nodeKey 关联（没用外键，因为 agent_logs
  ▎ 是高频写入的日志表，不想加外键约束影响写入性能）。
  ▎
  ▎ node_executions 存的是最终结果（output、summary、status）；agent_logs 存的是过程（第几轮调了什么工具、参数是什么、结果是什么）。前端'运行'tab 的时间线就是读 agent_logs
  ▎ 渲染的。"

  "snapshot 和 changes 存在 node_executions 里不会太大吗？"

  ▎ "snapshot 是执行前所有文件内容的 JSON（{ 'src/app.ts': '内容...' }），确实可能很大。我通过三个手段控制：排除 node_modules/.git/dist（99% 的文件量）；单文件 > 512KB
  ▎ 不读内容；实际测试正常项目几十个源文件，snapshot 约几百 KB。
  ▎
  ▎ 如果是大项目（几千文件），MySQL 存几十 MB 的 JSON 不合理。改进方案是只快照 Agent 实际操作过的文件（从 agent_logs 提取 write_file 的路径列表），或者有 Git 时直接用 git diff
  ▎  替代。"

# "你这个项目用 Redis 了吗？哪些场景需要？"
  
  回答：

  ▎ "当前没有接 Redis，用内存或数据库替代。但有几个场景如果上生产必须接：
  ▎
  ▎ 1. Agent Loop 并发锁
  ▎
  ▎ 当前用 Node.js 进程内的 Map 存锁状态（runningAgents.set(key, true)）。单进程没问题，但如果 PM2 cluster 跑多进程，进程 A 的 Map 锁不住进程 B 的请求。
  ▎
  ▎ → 需要 Redis 分布式锁（SET key value NX EX 300），所有进程共享锁状态。
  ▎
  ▎ 2. SSE 连接状态 / 执行进度缓存
  ▎
  ▎ 当前 Agent 执行进度只通过 SSE 实时推送，断了就没了。如果用户刷新页面，要从数据库 agent_logs 表查历史。
  ▎
  ▎ → 可以用 Redis 缓存当前执行进度（HSET execution:{id} status running iteration 5），前端重连后从 Redis 快速恢复，不用查数据库。
  ▎
  ▎ 3. AI API 限流
  ▎
  ▎ 当前没有请求限流。多个用户并发触发 Agent Loop，每个每秒发 1 次 LLM 请求，容易触发 AI API 的 RPM 限制。
  ▎
  ▎ → Redis 令牌桶 / 滑动窗口限流（INCR api:rpm:{minute} + EXPIRE），超限时排队等待而不是直接失败。
  ▎
  ▎ 4. 文档处理任务队列
  ▎
  ▎ 当前知识库文档上传后在 HTTP 请求中同步处理（解析 + 切片 + Embedding）。如果 PDF 很大或并发上传多个文件，进程会被阻塞。
  ▎
  ▎ → Redis 作为 Bull/BullMQ 的 broker，文档处理改为后台队列任务。上传后立即返回，Worker 从队列消费异步处理。
  ▎
  ▎ 5. 知识库检索缓存
  ▎
  ▎ 同一段文本重复检索（如用户多次执行同一个 PRD 审核节点），每次都要调 Embedding API + Milvus 搜索。
  ▎
  ▎ → Redis 缓存检索结果（SET rag:{query_hash} results EX 3600），相同 query 一小时内直接返回缓存，省 Embedding 调用。"

  ---
  追问："你的并发锁用 Map 有什么问题？Redis 分布式锁怎么实现？"
  
  ▎ "Map 锁的问题：只在当前进程内有效。如果用 PM2 cluster 模式跑 4 个进程，用户请求可能落到不同进程，进程 A 的 Map 里有锁，进程 B 不知道 — 同一个节点被并发执行两次，workDir 
  ▎ 里文件互相覆盖。
  ▎
  ▎ Redis 分布式锁实现：
  ▎ SET agent_lock:{workflowId}:{nodeKey} {processId} NX EX 300
  ▎ - NX：只有 key 不存在时才设置成功（原子操作，谁先设谁拿到锁）
  ▎ - EX 300：300 秒自动过期（防止进程崩溃后锁永远释放不了）
  ▎ - 值存 processId：释放时验证是不是自己的锁（防止误删别人的锁）
  ▎
  ▎ 释放锁用 Lua 脚本保证原子性：
  ▎ if redis.call('get', key) == processId then redis.call('del', key) end
  ▎
  ▎ 为什么不用 Redlock：单节点 Redis 的 SET NX EX 对我的场景够用。Redlock 是多 Redis 节点的共识算法，复杂度高，我不需要那么强的一致性保证 — 
  ▎ 极端情况下两个进程同时执行同一节点（锁短暂失效），最多文件被覆盖一次，不会造成数据损坏。"

  ---
  追问："任务队列为什么选 Bull/Redis 不用 RabbitMQ？"
  
  ▎ "Bull（基于 Redis）的优势对我的场景：
  ▎
  ▎ 1. 零运维成本：Redis 本来就要接（做锁和缓存），Bull 直接复用同一个 Redis，不需要额外部署 RabbitMQ。
  ▎ 2. Node.js 原生：Bull 是 npm 包，和 NestJS 有官方集成（@nestjs/bull），几行代码就能用。
  ▎ 3. 轻量级：我只需要'上传文档 → 后台异步处理'这个简单的生产者/消费者模型，不需要 RabbitMQ 的交换机/路由/死信队列等复杂特性。
  ▎
  ▎ 如果后续需要多消费者、消息持久化保证、复杂路由（比如不同文件类型分发给不同 Worker），再考虑 RabbitMQ/Kafka。当前规模 Bull + Redis 完全够用。"

  追问："检索缓存的 key 怎么设计？怎么保证缓存和知识库数据一致？"
  
  ▎ "Key 设计：rag:{knowledgeBaseIds的hash}:{query的hash}
  ▎
  ▎ 用 MD5 或 SHA256 对 knowledgeBaseIds 数组排序后拼接 + query 文本做 hash，作为缓存 key。相同的知识库范围 + 相同的 query = 命中缓存。
  ▎
  ▎ 一致性保证：
  ▎ - 知识库更新时主动失效：用户上传新文档或删除文档后，DEL rag:{knowledgeBaseId}:*（删该知识库相关的所有缓存）。
  ▎ - TTL 兜底：设 1 小时过期。即使没主动删，最多 1 小时后缓存自动失效，下次检索拿最新结果。
  ▎ - 用户可手动刷新：前端'重新检索'按钮强制跳过缓存（请求头加 Cache-Control: no-cache，后端判断后不读 Redis）。
  ▎
  ▎ 这是最终一致性，不是强一致性。对 RAG 场景够用 — 知识库更新后最多 1 小时内检索到旧结果，不会导致业务错误（审核标准不会频繁变）。"
