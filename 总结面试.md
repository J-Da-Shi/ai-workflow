 1. "你的 Agent Loop 和 Cursor 有什么区别？为什么不直接用 Cursor？"

  ▎ "架构原理完全一样——while 循环 + tool_calls + 结果反馈。但定位不同：
  ▎
  ▎ Cursor 是交互式的，人坐在电脑前实时对话。我的是流水线式的，用户配好 prompt 就走了，Agent 自动完成一整块任务，完成后等人审批。
  ▎
  ▎ 为什么不直接接 Cursor/Claude Code API：第一，我需要平台层能力 — 多节点编排、审批流程、Git 自动化、知识库管理，这些 Cursor 没有。第二，我需要对底层模型有选择权 — 简单任务用
  ▎  DeepSeek 降成本，复杂任务可以切到 Claude。第三，自己实现后我对 Agent 的每一步都有完整控制：白名单命令、路径安全、上下文压缩策略都是我决定的，不是黑盒。
  ▎
  ▎ 当然如果团队资源充足，最优方案是保留平台层自研 + 底层引擎接 Claude Code API，两层各管各的。"

  2. "RAG 检索质量怎么保证？你怎么知道检索到的内容是对的？"

  ▎ "当前版本靠三层过滤：第一层是 Milvus 的向量相似度 top-k；第二层是 score 阈值（0.3 以下的丢弃）；第三层是 Prompt 中告诉 AI '仅基于参考资料回答，无关内容忽略'。
  ▎
  ▎ 但说实话，这不够严谨。我没有量化评估 — 没有标注 query-relevant_chunk 的测试集，不知道 Recall@5 是多少。这是下一步要做的：准备 20 个测试 query，标注期望命中的 
  ▎ chunk，跑一遍算召回率。
  ▎
  ▎ 另外我缺 Rerank。当前 top-k 完全靠 embedding 相似度排序，但 embedding 是无监督的，不一定和"审核相关性"对齐。加一个 Cross-Encoder reranker（如 
  ▎ bge-reranker）能显著提升精度，代价是每次检索多 100-200ms 延迟。
  ▎
  ▎ 还有一个我踩过的坑：阿里云的 Embedding API 每批最多 10 条，我一开始设 20 导致全部失败。这种限制没有写在文档显眼位置，是看报错才发现的。"

  3. "安全性怎么考虑的？AI 会不会搞破坏？"

  ▎ "我在三个层面做了防护：
  ▎
  ▎ 文件层：所有路径操作先过 safePath 函数 — path.resolve 解析掉 .. 后检查是否还在 workDir 内。AI 传入 ../../etc/passwd 这种路径会被直接拒绝。
  ▎
  ▎ 命令层：run_command 工具用白名单，只允许 build/test/lint 这几个预定义命令前缀。为什么用白名单不用黑名单 — 黑名单永远有遗漏，rm 可以写成 \rm、$(rm ...)、反引号嵌套，AI 
  ▎ 的创造力比你想象的强。白名单是唯一 100% 安全的方案。
  ▎
  ▎ Git 层：Token 不在 Agent Loop 中暴露（Agent 看不到 token），Git 操作是执行引擎的前后置钩子，不是 AI 调用的工具。这样 AI 不可能 force push 或删分支。
  ▎
  ▎ 还有一个我在设计时考虑但没做的：输出大小限制。如果 AI 写了一个 10MB 的文件（比如 base64 编码的图片），write_file 不会拒绝。生产环境应该加单文件大小限制（比如 
  ▎ 1MB）和单次执行总写入量限制。"

  4. "Diff 审批为什么不用 Git？你全目录扫描不会很慢吗？"

  ▎ "这是一个我刻意的 trade-off。
  ▎
  ▎ 用 Git diff 确实是最优方案 — Git 本身就是做版本对比的专业工具，git diff HEAD 一行命令搞定。但我的场景有个前提：不是所有节点都配了 Git。有些用户只想让 AI 
  ▎ 写几个文件看看效果，不想配仓库。
  ▎
  ▎ 如果是几千个文件的大项目确实会慢。我的改进方案是：如果配了 Git 就用 Git diff，没配 Git 才走快照。这样两种场景都覆盖，大项目走 Git 高效，小项目不依赖 Git 也能用。"

  5. "为什么不用 LangChain？"

  ▎ "我认真评估过 LangChain。它的价值在于：统一抽象多种 LLM / 向量库 / 检索策略，快速实验不同配置。
  ▎
  ▎ 但我最终没用，三个原因：
  ▎
  ▎ 第一，我的链路很短。解析 → 切片 → Embedding → 存 Milvus → 搜索，每一步直接调对应的库（pdf-parse / OpenAI SDK / Milvus SDK）总共 100 行代码。LangChain 在每一步上包 3-4
  ▎ 层抽象，出了问题我要翻好几层源码 debug。
  ▎
  ▎ 第二，依赖膨胀。LangChain 的 node_modules 有几十 MB，拖了一堆我用不到的模块。我只装了 3 个包。
  ▎
  ▎ 第三，生产环境需要控制力。比如 Embedding 批量大小我需要设成 10（阿里云限制），重试逻辑我需要指数退避。这些在 LangChain 中要 override 它的默认行为，不如自己写 20
  ▎ 行代码清晰。
  ▎
  ▎ 如果后续要加 Rerank、混合检索（BM25 + 向量）、多轮对话记忆，会考虑引入 LangChain 的特定模块而不是全量依赖。"

  6. "工作流执行中某个节点挂了怎么办？你有容错机制吗？"
  
  ▎ "当前的容错分两层：
  ▎
  ▎ 节点级：每个节点执行都在 try/catch 中，失败时 execution.status 设为 'failed'，error 字段记录错误信息。工作流不会因为一个节点失败而整体崩溃 — 
  ▎ 它会停在失败的节点，用户可以看到报错，修改 Prompt 后重新执行。
  ▎
  ▎ Agent Loop 级：设了 200 轮上限防止死循环；并发锁（Map）防止同一节点被重复触发；工具执行异常被 catch 后返回错误文本给 AI，AI 可以自行调整策略而不是整个 Loop 崩掉。
  ▎
  ▎ 但有几个我知道的不足：第一，没有重试机制 — 如果 DeepSeek API 偶尔超时，当前直接失败。生产级应该对 5xx 错误自动重试 2-3 次。第二，没有断点续跑 — 工作流跑到第 3 
  ▎ 个节点失败后，修复后只能从第 3 个节点重跑，不能从整体重跑（会重复执行已通过的节点）。这个目前通过审批机制缓解 — 已通过的节点状态是 'approved'，executeWorkflow 遇到 
  ▎ approved 的节点会跳过。第三，没有超时熔断 — 如果 Agent Loop 跑了 30 分钟还没结束（比如 AI 进入了低效循环），当前没有外部 kill 机制，只能等它自己到 200 轮上限。"

  7. "你的系统能支持多少并发用户？瓶颈在哪？"
  
  ▎ "当前架构的瓶颈在 AI API 调用。单个 Agent Loop 执行可能跑 10-20 轮，每轮一次 LLM 调用（1-5 秒），一个节点执行可能要 30-60 秒。这期间占用一个连接。
  ▎
  ▎ 并发能力取决于几个因素：
  ▎
  ▎ CPU/内存：NestJS 是单进程 Node.js，但 I/O 是异步的（await LLM API），所以并发请求不会互相阻塞。10 个用户同时执行节点，10 个 Agent Loop 并行跑没问题 — 瓶颈不在服务端计算。
  ▎
  ▎ AI API 限流：DeepSeek/GPT 都有 RPM（每分钟请求数）限制。10 个用户同时跑 Agent Loop，每个每秒发 1 次请求 = 600 RPM，容易触发限流。解决方案是请求队列 + 限流器。
  ▎
  ▎ Milvus：Zilliz Cloud Serverless 自动扩缩，搜索延迟 < 100ms，不是瓶颈。
  ▎
  ▎ MySQL：几十个并发查询完全没问题，不是瓶颈。
  ▎
  ▎ 如果要生产部署，我会加三个东西：1. PM2 cluster 模式跑多进程 2. AI API 调用加请求队列和限流 3. 长时间执行的 Agent Loop 改为后台任务（Bull queue），HTTP 请求立即返回，用 
  ▎ WebSocket/SSE 推进度。"

  8. "你怎么保证节点之间的数据一致性？比如 PRD审核通过了但代码开发节点引用的是旧版 PRD？"
  
  ▎ "当前设计是快照式的 — 每个节点执行时从上游拿到的是那一刻的 output，不会因为上游后来重新执行而自动更新。
  ▎
  ▎ 具体来说：代码开发节点的 getNodeInput 读取 PRD审核节点的 execution.output。如果用户后来重新执行了 PRD审核（生成了新的审核报告），代码开发节点不会自动感知 — 
  ▎ 除非用户手动重新执行代码开发节点。
  ▎
  ▎ 这是有意的设计：工作流是有序执行的，每个节点只在自己执行时读取上游的最新输出。如果上游重跑了，下游节点应该由用户决定是否重跑，而不是自动级联执行（可能导致代码被覆盖）。
  ▎
  ▎ 改进方向是加版本标记：每次节点执行记录一个版本号，下游执行时记录"我引用的是上游的第 N 版输出"。如果上游更新了，前端提示"上游已更新，是否重新执行"。"

  9. "切片策略为什么选 512 token？有没有做过对比实验？"
  
  ▎ "512 token 是参考 OpenAI 官方 cookbook 和 LlamaIndex 的实验结论。他们测试了 128/256/512/1024/2048 五个粒度，在 QA 任务上 512 的 Recall@5 和 Answer Quality 综合最优。
  ▎
  ▎ 原理是这样：切太小（128）→ 每个 chunk 信息量不够，搜到了也没用；切太大（2048）→ 向量代表的语义太模糊，不同话题混在一个向量里，检索精度下降。512 是 sweet spot。
  ▎
  ▎ 但 512 不是银弹。如果文档结构很强（比如每个章节讨论一个独立话题），按章节切更合理，不管它是 300 还是 800 token。我的实现里已经做了"Markdown 按标题优先分割" — \n\n 
  ▎ 是主要分割点，段落不超过 chunkSize 就合并，超了才硬切。
  ▎
  ▎ 实话说我没做对比实验。如果要做，方法是：准备 50 个测试 query + 标注答案所在的文档段落，用不同 chunkSize 分别入库，跑 Recall@5 对比。这是后续优化的方向。"

  10. "SSE 连接断了怎么办？比如用户网络抖动？"
  
  ▎ "当前没有处理这个情况 — SSE 连接断开后，前端 reader.read() 会 reject，进 catch 块显示'执行失败'。但实际后端 Agent Loop 
  ▎ 还在跑（不会因为前端断开就停），只是用户看不到进度了。
  ▎
  ▎ 解决方案有两个层面：
  ▎
  ▎ 前端：catch 后可以 fallback 到轮询模式 — 每 3 秒调 GET /executions 查状态，如果节点还在 running 就等，变成 waiting/failed 就停止轮询。这样即使 SSE 
  ▎ 断了，用户刷新页面后还能看到结果。
  ▎
  ▎ 后端：Agent Loop 的结果本来就存在数据库（execution.output），不依赖 SSE 连接。SSE 只是实时展示用的，最终结果在数据库。用户刷新页面后，nodeRun 组件会从 GET /agent/logs 
  ▎ 加载历史日志 — 这就是为什么我同时做了日志持久化（agent_logs 表）和 SSE 实时推送，两条路互为备份。
  ▎
  ▎ 目前的不足是：重连后看不到中间过程（错过的 thinking/tool_done 事件没了）。改进方向是 SSE 连接断开后，前端用 GET /agent/logs?since=lastEventTime 补偿缺失的事件。"

  11. "如果 AI 生成的代码有安全漏洞怎么办？你怎么保证代码质量？"
  
  ▎ "这是一个很好的问题，我从三个层面考虑：
  ▎
  ▎ 第一层：Prompt 约束。System Prompt 里明确告诉 AI"避免 SQL 注入、XSS、硬编码密钥"等。但 Prompt 约束是 soft 的，AI 不一定 100% 遵守。
  ▎
  ▎ 第二层：自动验证。run_command 工具让 Agent 写完代码后跑 npx tsc --noEmit（类型检查）和 npm run lint（规范检查）。如果 lint 规则里配了安全相关规则（如 
  ▎ eslint-plugin-security），能自动检出一部分漏洞。
  ▎
  ▎ 第三层：人工审批。这是最关键的 — Diff 审批机制确保所有 AI 生成的代码在合并前都经过人工 review。前端展示完整的代码 diff，reviewer 可以逐文件检查。审批驳回时代码自动还原。
  ▎
  ▎ 但说实话，这三层都不能 100% 保证代码安全。生产级的做法应该是：在 Git PR 创建后，接入公司的 SAST（静态安全扫描）工具自动扫描，有高危漏洞时阻止合并。这属于 CI/CD 
  ▎ 层面的能力，不在我的平台范围内，但和我的 Git 集成天然兼容 — PR 创建后公司的 CI pipeline 自动触发。"

  12. "为什么节点间传递用 JSON 字符串而不是结构化数据库字段？"
  
  ▎ "设计决策有两个考虑：
  ▎
  ▎ 兼容性：output 字段原来是 TEXT 类型存纯文本。如果改成 JSON 字段，需要数据库 migration（ALTER TABLE），已有的执行记录全部不兼容。用 JSON.stringify 存在 TEXT 
  ▎ 字段里，旧数据（纯文本）还能用 — getNodeInput 里 try JSON.parse catch 后 fallback 返回原文。零 migration 成本。
  ▎
  ▎ 灵活性：不同节点类型的输出结构不同（code_dev 有 changedFiles，prd_review 有 verdict）。如果用强 schema（多个字段），每加一种节点类型就要 ALTER TABLE。JSON 字符串是 
  ▎ schema-less 的，随时加字段不影响表结构。
  ▎
  ▎ 缺点我知道：不能用 SQL 直接查结构化字段（比如 WHERE verdict = 'fail'）。如果后续需要按审核结论统计，要么解析 JSON，要么加冗余字段。当前规模不需要，所以选了更灵活的方案。"

  13. "embedding 模型为什么不和 Chat 模型用同一个服务？"
  
  ▎ "因为我的 Chat 模型用的是中转站（支持 GPT-5.4/Claude 等），但中转站没有 Embedding 模型。Embedding 用的是阿里云百炼的 text-embedding-v3。
  ▎
  ▎ 这是实际项目中很常见的情况 — Chat 和 Embedding 用不同的服务商。好处是解耦：Chat 模型可以随时从 GPT 换成 Claude，不影响 Embedding；Embedding 提供商挂了不影响对话功能。
  ▎
  ▎ 代码层面的实现：EmbeddingService 有自己独立的 EMBEDDING_API_KEY 和 EMBEDDING_BASE_URL 环境变量，和 Chat 用的 AI_API_KEY / AI_BASE_URL 完全隔离。两个 OpenAI 
  ▎ 客户端实例互不干扰。
  ▎
  ▎ 维度是 1024（text-embedding-v3），Milvus Collection 的 FloatVector dim 也是 1024。如果换 Embedding 模型（比如换成 OpenAI 的 text-embedding-3-small = 1536 维），需要重建 
  ▎ Collection + 重新 embedding 所有文档。这是一次性迁移成本。"

  14. "工作流执行的审批机制，如果审批人不在线怎么办？"
  
  ▎ "当前设计是同步阻塞式审批 — 节点执行完进入 waiting 状态，整个工作流暂停，直到有人点通过或驳回。
  ▎
  ▎ 这在当前场景（个人使用/小团队）是合理的。但如果是大团队生产环境，确实需要考虑：
  ▎
  ▎ 1. 超时自动通过：节点配置中加一个"审批超时时间"，超过 N 小时无人审批就自动通过（或自动驳回，取决于策略）
  ▎ 2. 多人审批：当前只有一个审批人（触发执行的用户）。生产级应该支持指定 reviewer 列表，任意一人审批即可
  ▎ 3. 通知机制：节点进入 waiting 后，通过钉钉/Slack/邮件通知审批人。当前没有集成任何通知渠道
  ▎ 4. 可跳过审批：节点配置有 requireApproval 开关，设为 false 的节点会自动 approve 继续下一个，不需要人工
  ▎
  ▎ 我已经做了第 4 点 — executeWorkflow 中检查 nodeConfig.requireApproval，如果为 false 就自动调 approveNode 继续后续节点。"

  15. "Milvus 的索引策略怎么选的？为什么用 IVF_FLAT 不用 HNSW？"
  
  ▎ "IVF_FLAT 和 HNSW 都是常用的 ANN（近似最近邻）索引，各有适用场景：
  ▎
  ▎ IVF_FLAT：先把向量聚类成 N 个桶（我设了 nlist=128），搜索时只扫描最近的几个桶（nprobe=16）。优点是内存占用小（只多存聚类中心），缺点是精度比 HNSW 低。
  ▎
  ▎ HNSW：建一个多层图结构，搜索时沿图边跳转到最近的节点。优点是精度最高、搜索最快，缺点是内存占用是原始数据的 2-3 倍。
  ▎
  ▎ 我选 IVF_FLAT 是因为：当前数据量 < 1 万条向量（几百个文档 × 几十个切片），两种索引的精度差异可以忽略。IVF_FLAT 内存占用低，适合 Zilliz Cloud 免费版的资源限制。
  ▎
  ▎ 如果数据量涨到百万级，我会换 HNSW 或 IVF_PQ（量化压缩 + 倒排，平衡内存和精度）。
  ▎
  ▎ 度量方式选 COSINE 不选 L2（欧氏距离）是因为文本 embedding 的"方向"比"长度"更有意义 — 两段语义相近的文本向量指向相似的方向，但长度可能因为文本长短不同而差异很大。COSINE 
  ▎ 归一化了长度只比方向。"

  AI Dev Workflow — AI 驱动的软件开发工作流编排平台

  项目简介

  面向研发团队的 AI 工作流编排平台，将需求分析、PRD审核、代码开发等环节编排为自动化流水线，通过 AI Agent 自主编码 + RAG 知识库审核 + Git
  自动化提交，减少重复性人工操作，提升研发流程标准化程度。

  技术栈

  React 19 + ReactFlow + Ant Design + react-diff-viewer NestJS + TypeORM + MySQL + Milvus（向量检索） OpenAI 兼容 API + text-embedding-v3 | SSE + simple-git + Zilliz Cloud

  项目职责

  1. 设计并实现 AI Agent 编码引擎：基于 Tool Use 协议实现 Agent Loop（文件读写/代码搜索/命令执行），
  支持写完代码自动编译验证 + 失败自愈（最多 3 轮修复），单节点可自主完成 10+ 文件的创建与修改，替代人工逐文件编写，将重复性编码工作耗时从小时级降至分钟级
  2. 落地 RAG 知识库驱动的 PRD 审核：完成文档处理全链路（解析 → 递归切片 → 批量 Embedding → Milvus
  入库），审核时自动检索相关规范作为参考依据，使审核标准从主观判断转为可量化的结构化评分，覆盖文档完整性/技术可行性/边界条件等 4 个维度
  3. 实现 Git 全流程自动化：封装 clone/branch/commit/push/PR 创建逻辑，适配 GitHub + GitLab 双平台，审批通过后自动推送代码并创建
  PR，规避手动操作导致的分支命名混乱、忘记提交等问题，保障代码交付规范性
  4. 设计上下文管理与安全防护机制：通过滑动窗口压缩 + 源头截断控制 Agent 上下文膨胀（支持 200 轮长任务不超窗口）；路径穿越校验 + 命令白名单确保 AI
  操作可控，杜绝越权文件访问和危险命令执行
  5. 实现 SSE 实时推送与 Diff 审批流程：Agent 执行过程实时可见（思考/工具调用/结果），完成后展示代码 Diff 供人工审查，驳回可一键还原，确保 AI 产出 100% 
  经过人工确认后才进入代码库
  6. 打通多节点自动串联执行：设计节点间结构化 JSON 协议（类型标识 + 内容 + 扩展字段），实现拓扑排序 + 审批控制 +
  自动续跑，支持"需求→审核→开发"全链路无人值守执行，仅在关键节点暂停等待人工审批
