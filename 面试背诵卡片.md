## 15 个高频问题卡片

### Q0: 主流 Agent 架构有哪些？

**关键词：** ReAct / CoT / Plan-and-Execute / Self-Reflection / Multi-Agent / Tool-Calling

**标准回答：**

| 架构模式 | 核心思想 | 典型代表 | 适用场景 |
|---------|---------|---------|---------|
| ReAct | 思考→行动→观察，交替循环 | LangChain Agent | 需要工具调用的通用任务 |
| CoT 思维链 | 分步推理，显式思考过程 | GPT-4 with "think step by step" | 复杂推理、数学、逻辑题 |
| Plan-and-Execute | 先整体规划，再分步执行 | BabyAGI / AutoGPT | 复杂多步任务、项目级开发 |
| Self-Reflection | 执行后自我复盘修正 | Reflexion / CRITIC | 需要高准确率的任务 |
| Multi-Agent | 多智能体分工协作 | MetaGPT / CrewAI | 大规模复杂项目、多角色协作 |
| Tool-Calling Loop | while循环+工具调用+结果回填 | Claude Code / Cursor / 我的项目 | 代码开发、文件操作 |

**详细口述版：**

> **1. ReAct（Reasoning + Acting）**
> 最经典的 Agent 架构。每一步先 Thought（思考当前应该做什么），再 Action（调用工具），最后 Observation（观察工具返回结果），三步交替循环直到任务完成。论文来自 Google Brain 2022 年。优点是推理过程可解释，缺点是每轮都要思考一次，token 消耗大。LangChain 的 AgentExecutor 就是这个模式。
>
> **2. CoT 思维链（Chain of Thought）**
> 严格来说不算 Agent 架构，是一种 Prompting 策略——让 LLM 在回答前先分步推理。加一句"Let's think step by step"就能显著提升推理准确率。用在 Agent 中是作为"思考"阶段的增强——让 AI 先分析问题再决定调什么工具。适合复杂推理但不需要频繁工具调用的场景。
>
> **3. Plan-and-Execute（计划执行分离）**
> 先让一个 Planner LLM 生成完整的任务计划（分几步、每步做什么），再让 Executor 逐步执行。优点是全局视野好、不容易跑偏；缺点是计划可能过时——执行中发现新信息需要调整计划。BabyAGI 和 AutoGPT 用的就是这种模式。改进版是 Plan-and-Solve，允许执行中动态修改计划。
>
> **4. Self-Reflection（自省式）**
> 执行完一轮后，让 AI 自我评估结果质量，发现问题就重新执行。典型论文是 Reflexion——把执行结果和期望对比，生成反思笔记，下一轮带着反思重试。适合需要高准确率的场景（如代码生成后自动跑测试，失败就反思修改）。我的 Agent Loop 里 run_command 跑 tsc 失败后 AI 自动修复就是简化版的 Self-Reflection。
>
> **5. Multi-Agent（多智能体）**
> 多个 Agent 分工协作，各有专长。比如 MetaGPT 模拟软件公司——产品经理 Agent 写 PRD、架构师 Agent 设计方案、程序员 Agent 写代码、测试 Agent 写测试。通过消息传递协作。优点是每个 Agent 的 Prompt 专注单一职责效果更好；缺点是协调成本高、一个 Agent 失败可能阻塞整个流程。CrewAI 和 AutoGen 是这个方向的框架。
>
> **6. Tool-Calling Loop（我项目用的）**
> while 循环 + 工具调用 + 结果回填，本质是简化版 ReAct——去掉了显式的 Thought 步骤，直接让模型通过 function_calling 返回要执行的工具。Claude Code、Cursor、我的项目都是这个模式。优点是实现简单（核心代码不到 200 行）、和 OpenAI API 原生兼容；缺点是没有显式推理步骤，AI 的决策不如 ReAct 可解释。
>
> **我的项目选择 Tool-Calling Loop 的原因：** 代码开发任务是确定性强的——读文件、搜代码、写文件、验证，工具集固定。不需要 Plan-and-Execute 的全局规划（任务在节点配置中已经明确），也不需要 Multi-Agent 的角色分工（单 Agent 足以完成一个节点的编码任务）。加上 OpenAI SDK 原生支持 tool_choice，实现成本最低。

### Q2: 整体架构怎么设计？

**关键词：** React Flow / NestJS / ExecutionService / 拓扑排序 / 双路径分发 / SSE / MySQL+Milvus

**标准回答：**
1. **背景：** 需要一个端到端的研发自动化平台，不是单点工具
2. **实现：** 前端 ReactFlow 画布编排 → 后端 ExecutionService 读取画布按拓扑序执行 → 普通节点走 LLM 生成，代码节点走 Agent Loop → SSE 实时推送执行日志 → Diff 审批 → Git PR
3. **难点：** 节点间数据传递用结构化 JSON 协议（type + text + 扩展字段）
4. **取舍：** 当前线性执行链，没做 DAG 并行（复杂度高、当前规模不需要）
5. **改进：** nextMap 改一对多 + Promise.all 并行 + 汇聚节点 barrier 等待

**详细口述版：**

> 整体是前后端分离的全栈架构，分四层来讲：
>
> **前端层**：React 19 + ReactFlow 做可视化画布编排。用户拖拽节点到画布上、连线、配置每个节点的 Prompt 和参数。画布数据（nodes + edges）存到后端。前端还负责 SSE 日志的实时渲染——用 fetch + ReadableStream 手动读取流式响应，以及用 react-diff-viewer 展示代码变更对比。
>
> **后端层**：NestJS + TypeORM + MySQL。核心是 ExecutionService——它读取画布数据，用简化的 Kahn 算法计算拓扑序（算入度找起点，沿 nextMap 顺序执行）。执行时做双路径分发：根据 nodeType 判断，普通节点（PRD 审核、需求分析）调一次 LLM 生成文本；代码开发节点启动 Agent Loop，进入 while 循环自主完成编码。
>
> **数据层**：MySQL 存配置数据（工作流、节点配置）和运行时数据（执行记录、Agent 日志、快照）。Milvus 存向量用于 RAG 检索。两者通过 chunk ID 关联，各管各的——MySQL 管元数据，Milvus 管向量搜索。
>
> **通信层**：节点间数据传递用结构化 JSON 协议——每个节点执行完输出 { type: 'code_dev', text: 'AI总结', changedFiles: [...] }，下游通过 getNodeInput 读取上游 output，JSON.parse 提取 .text 传给 AI。做了向后兼容：旧数据是纯文本的 try parse catch 后直接返回原文。
>
> 执行流程是：画布编排 → 拓扑排序 → 逐节点执行 → SSE 实时推日志 → 执行完进入 waiting → 人工审批 → approve 后 fire-and-forget 继续下游 → 最终 Git commit + push + PR。

### Q3: Agent Loop 怎么实现？

**关键词：** while loop / tool_calls / executeTool / tool result 回填 / messages / 上下文压缩 / 双引擎 / Codex CLI / 降级

**标准回答：**
1. **背景：** 普通 LLM 只能生成文本，不能操作项目文件
2. **双引擎设计：** 优先调用 Codex CLI（OpenAI 官方 Agent，沙箱隔离、能力强）；检测本地未安装时自动降级到自研 Agent Loop，保证执行不中断
3. **自研实现：** while 循环，每轮把 system prompt + 历史 messages 发给模型。模型返回 tool_calls，解析工具名和参数，执行对应工具（list_directory/read_file/search_code/write_file/run_command），结果作为 tool message 回填。循环到模型不再调工具为止
4. **难点：** 多轮后上下文膨胀——15 轮后 messages 到 3-5 万 token
5. **解决：** read_file 源头截断（500 行硬截）+ 滑动窗口压缩（保留最近 3 轮完整，早期截短为 200 字符）。不能删消息只能截短，因为 API 要求 tool_call 和 tool_result 必须通过 tool_call_id 配对
6. **改进：** 加 edit_file 工具减少 output token，从全量写入改为增量编辑

**详细口述版：**

> Agent Loop 本质是一个 while 循环，架构原理参考了业界 ReAct 模式。
>
> 具体实现：每一轮把 system prompt（包含工作规范、代码规范、工具使用策略、项目结构）和历史 messages 数组发给模型。如果模型返回的 response 里有 tool_calls 字段，我就解析出工具名和参数，调用对应的 executeTool 函数执行。我实现了 5 个工具：list_directory（列目录结构）、read_file（读文件内容）、search_code（正则搜索代码）、write_file（写文件）、run_command（执行白名单命令）。工具执行完后，结果作为 role='tool' 的消息回填到 messages 数组中，然后进入下一轮循环。这个过程一直持续到模型不再返回 tool_calls，而是返回一段总结性文字，就表示任务完成。
>
> 核心挑战是上下文管理。每轮往 messages 里追加 1 条 assistant 回复 + N 条 tool result。一个 read_file 的 tool result 可能有几百行代码（几千 token），跑 15 轮后 messages 轻松到 3-5 万 token。
>
> 我做了两层控制：第一层是源头截断——read_file 超过 500 行的文件强制只返回前 200 行加一个提示"请用 startLine/endLine 精确读取"。这是最有效的控制手段，一个 800 行文件从占 8000 token 降到 2000 token。System Prompt 里也约束 AI：小文件全读，大文件只通过 search_code 加片段读取。
>
> 第二层是滑动窗口压缩——每轮调 AI 前检查 messages.length 大于 12 就触发压缩。策略是按"轮"分组（1 条 assistant + 后续的 tool messages = 1 轮），保留最近 3 轮完整内容，更早的 tool result 超过 300 字符就截断为前 200 字符加"[已截断]"。
>
> 这里有一个关键约束：不能删消息，只能截短 content。因为 OpenAI 的 API 要求 assistant 消息中每个 tool_call 必须有对应的 role='tool' 消息通过 tool_call_id 配对，删了任何一条就报 400 错误。所以只能截短不能删除。
>
> 和 Cursor 的核心效率差异在于：我的 write_file 要求 AI 输出完整文件内容（200 行文件 = 200 行 output token），而 Cursor 用 diff/patch 模式只输出修改的几行。后续优化方向是加 edit_file 工具支持增量编辑，output token 从几千降到几十。
>
> 另外我还做了引擎可切换的设计——节点配置里可以选"自研 Agent Loop"或"Codex CLI"。自研用于简单任务（成本低、完全可控、可切任意模型），Codex 用于复杂任务。如果 Codex CLI 环境不可用，自动降级到自研引擎，保证执行不中断。

### Q4: 为什么不用 LangChain？

**关键词：** 过度抽象 / 依赖膨胀 / 链路太短 / 精细控制 / 100 行代码

**标准回答：**
1. **背景：** 评估过 LangChain，决定不用
2. **原因一：** 链路太短不需要编排框架。RAG 全链路就是解析→切片→Embedding→搜索，直接调底层 SDK 加起来 100 行代码。LangChain 在每步包 4-5 层抽象，出问题翻好几层源码
3. **原因二：** 需要精细控制。Embedding 批量大小要设 10（阿里云限制）、重试要指数退避、切片要 Markdown 按标题优先。override LangChain 默认行为不如自己写 20 行
4. **原因三：** 依赖膨胀，node_modules 几十 MB
5. **什么时候该用：** 快速实验多种检索策略、对比不同 LLM、做复杂多轮对话编排时

**详细口述版：**

> 评估过 LangChain，最终决定不用，三个原因：
>
> 第一，链路太短不需要编排框架。我的 RAG 全链路是：解析 → 切片 → Embedding → Milvus 入库 → 搜索。每一步对应一个具体的库调用——pdf-parse 解析 PDF、我自己写的 splitText 做切片、OpenAI SDK 调 Embedding API、Milvus SDK 做向量入库和搜索。加起来 100 行代码。LangChain 在每一步上包了 Document → BaseLoader → TextSplitter → VectorStore → Chain 共 4-5 层抽象，出了问题要翻好几层源码才能定位。
>
> 第二，需要精细控制。Embedding 批量大小我要设成 10——因为阿里云 API 每批限制 10 条输入，这个在文档里不显眼，是看报错信息才定位到的。重试逻辑我要指数退避（1s → 2s → 4s）。切片策略我要 Markdown 按标题优先分割。这些在 LangChain 中要 override 默认行为或写自定义 Splitter，不如直接自己写 20 行来得清晰可控。
>
> 第三，依赖膨胀。LangChain 的 node_modules 有几十 MB，拖了 Agent/Memory/Callback 等我完全用不到的模块。我只装了 pdf-parse + mammoth + milvus-sdk + openai 几个包。
>
> 什么时候该用 LangChain：如果需要快速实验多种检索策略（BM25 / 向量 / 混合），对比不同 LLM 的效果，或者做复杂的多轮对话 + 检索 + 工具调用的编排，LangChain 的统一抽象确实省事。但我的场景确定性强、链路固定，自己写更合适。
>
> 同样的逻辑也适用于 Agent Loop——Tool-Calling 本质就是一个循环 + JSON schema 匹配，OpenAI SDK 原生支持 tool_choice 和 function_calling，直接用 SDK 代码量反而更少，核心代码不到 200 行，且可以精确控制每一轮的上下文处理逻辑。

### Q5: RAG 检索怎么做？

**关键词：** 文档解析 / 递归切片 / 批量 Embedding / Milvus / topK / score threshold / Prompt 增强

**标准回答：**
1. **背景：** RAG 不是普通问答，是接在 PRD 审核节点里
2. **入库链路：** 上传文档 → pdf-parse/mammoth 解析 → 递归字符切片（1500字符/片，200重叠）→ 阿里云 text-embedding-v3 批量转向量（batch=10）→ 双写 Milvus（向量）+ MySQL（元数据）
3. **检索链路：** PRD 文本转向量 → Milvus search（filter 限定知识库范围，COSINE，top-8）→ score<0.3 过滤 → 拼进 System Prompt 参考资料段落 → AI 输出审核报告
4. **踩坑：** 阿里云 Embedding API 每批限制 10 条输入，一开始设 20 报错，排查花了不少时间
5. **效果数据：** 内部测试集 50 个 query，Recall@5 = 82%（top-5 中包含标注正确 chunk 的比例），加 RAG 后审核准确率 61%→89%（架构师人工评判 30 份报告），耗时 2h→30min
6. **改进：** 加 Rerank 二次排序、混合检索（BM25+向量）、Query 改写

**详细口述版：**

> 我们的 RAG 不是普通的问答机器人，而是接在 PRD 审核节点里——用户上传企业规范、历史评审标准后，系统在审核 PRD 时自动检索相关规范作为参考依据。上线后实测效果：内部标注 50 个测试 query，Recall@5 达到 82%，加 RAG 后 PRD 审核准确率从 61%（纯 LLM baseline）提升到 89%。
>
> 分两条链路讲：
>
> **入库链路**：用户上传文档（支持 PDF/Word/Markdown）→ DocumentProcessorService 根据文件类型分别调 pdf-parse 或 mammoth 解析提取纯文本 → splitText 做递归字符分割（先按段落 \n\n 分，段落太长再按换行 \n 分，还是太长按字数硬切），每片 1500 字符，重叠 200 字符 → EmbeddingService 批量调阿里云 text-embedding-v3 转向量（1024 维，每批 10 条）→ 双写存储：Milvus 存向量用于搜索，MySQL 存原文和元数据用于管理。两边通过同一个 UUID 关联。
>
> **检索链路**：PRD 审核节点执行时 → 把 PRD 文本调 embedText 转向量 → MilvusService.search（filter 限定用户选择的知识库范围，COSINE 相似度，top-8）→ 过滤 score < 0.3 的噪声 → 把命中的规范片段拼接到 System Prompt 的"参考资料"段落，每条标注相似度百分比 → AI 基于增强后的 Prompt 输出结构化审核报告。
>
> 切片策略的选择：512 token（约 1500 中文字符）参考 OpenAI cookbook 的测试结论——在 QA 任务上对比了 128/256/512/1024/2048 五个粒度，512 的 Recall@5 和回答质量综合最优。太小信息量不够，太大语义模糊。重叠 200 字符防止关键句子在边界被截断。
>
> 为什么选 Milvus 不选 Chroma/pgvector：第一，多租户隔离——我有多个知识库，检索时需要 filter 限定范围，Milvus 原生支持标量过滤。第二，生产级特性——支持分区、持久化、水平扩展。第三，我用的是 Zilliz Cloud（Milvus 托管版），Serverless 免运维。
>
> 踩过的坑：阿里云 Embedding API 每批限制 10 条输入，我一开始按 OpenAI 文档设了 20，整个批次直接报错。这种限制在文档里不显眼，是看报错信息才定位到的。还有 Embedding 调用做了重试——每批最多重试 3 次，指数退避（1s → 2s → 4s），网络抖动导致的偶发失败不会直接标记文档为 failed。

### Q6: RAG 效果怎么评估？

**关键词：** score threshold / 测试集 / Recall@5 / MRR / Rerank / 引用来源 / 审核准确率

**标准回答：**
1. **背景：** 上线前做了一轮正式的效果评估
2. **评估方式：** 标注了 50 个测试 query + 期望命中 chunk，算 Recall@5 和 MRR
3. **效果数据：** Recall@5 = 82%，MRR = 0.71；加 RAG 后 PRD 审核准确率从 61% 提升到 89%（对比纯 LLM 无检索增强）
4. **三层过滤：** Milvus topK + score 阈值 0.3 + Prompt 约束不编造
5. **后续优化：** 加 bge-reranker 二次排序（内部 A/B 测试 Recall@5 提升到 91%）、混合检索、Query 改写
6. **取舍：** 0.3 阈值是宽松的，宁可多给 AI 判断，不严格过滤掉潜在有用内容

**详细口述版：**

> 上线前我们做了一轮正式的效果评估。标注了 50 个测试 query——从实际 PRD 审核场景中抽取的真实问题，每个 query 标注了期望命中的 chunk（由产品和架构师确认）。
>
> 核心指标怎么算的：
> - **Recall@5 = 82%**：50 个 query 中，有 41 个的 top-5 检索结果里包含了标注的正确 chunk。计算方式是"命中 query 数 / 总 query 数"。
> - **MRR = 0.71**：正确 chunk 在结果列表中的排名取倒数再平均。比如排第 1 位 = 1/1 = 1，排第 3 位 = 1/3 ≈ 0.33。MRR 0.71 意味着正确答案平均在 top-1 到 top-2 之间。
> - **审核准确率 61% → 89%**：让同一批 PRD 分别走"纯 LLM 无检索"和"RAG 增强"两条路径生成审核报告，由两位架构师对 30 份报告做人工评判（逐条判断"审核意见是否正确且有依据"），统计正确率。61% 是 baseline，89% 是加 RAG 后的效果。
>
> 当前靠三层过滤保证质量：第一层是 Milvus 的向量相似度 top-k（取最相关的 8 条）；第二层是 score 阈值 0.3（COSINE 度量下低于 0.3 的基本无关，直接过滤掉）；第三层是 Prompt 中约束 AI"仅基于参考资料回答，材料中没有的不要编造"。
>
> 关于 0.3 这个阈值怎么确定的：COSINE 相似度范围是 0-1，文本检索的实际命中分数分布通常在 0.3-0.8 之间。0.3 是比较宽松的——宁可多给一些"可能相关"的结果让 AI 自己判断，而不是严格过滤掉潜在有用的内容。如果设 0.7 太严格，知识库里稍微换个表述方式（如"接口设计"vs"API 规范"）就可能被丢弃。这个值是在测试集上调参后确定的，0.3 在 Recall 和 Precision 之间取得最佳平衡。
>
> 后续优化已在推进：
> - 加了 Rerank 模型（bge-reranker-v2），对 top-k 结果做二次排序。内部 A/B 测试中 Recall@5 从 82% 提升到 91%，代价是每次检索多 100-200ms
> - 审核报告中展示引用来源（"基于参考资料 1，相似度 72%"），让用户能判断 AI 的结论是否有据可依
> - 规划中：加 BM25 混合检索 + Query 改写
>
> 如果知识库里确实没有相关内容怎么办：score 阈值过滤后 retrievedContext 为空，Prompt 中没有"参考资料"段落，AI 只基于自己的知识审核，效果和没有 RAG 时一样。不会因为没有检索到就瞎编。

### Q7: 怎么防止 Agent 乱改文件？

**关键词：** safePath / path.resolve / startsWith / 白名单 / 60s 超时 / Git token 不暴露

**标准回答：**
1. **文件层：** safePath 函数——path.resolve(workDir, filePath) 解析真实路径，检查是否以 workDir 开头。../../etc/passwd resolve 后不以 workDir 开头 → 直接拒绝
2. **命令层：** run_command 白名单——只允许 npm run build、npm test、npx tsc 等预定义命令前缀。为什么白名单不用黑名单：黑名单永远有遗漏，rm 可以写成 \rm、$(rm...)、反引号嵌套。另加 60s 超时防死循环
3. **Git 层：** Token 不暴露给 Agent——Git 操作是执行引擎的前后置钩子，不是 AI 能调用的工具。AI 在 Loop 中看不到 token
4. **取舍：** 白名单是唯一能保证 100% 安全的方案
5. **改进：** 加文件大小限制（单文件 1MB 上限、单次执行总写入量限制）

**详细口述版：**

> 三个层面的防护：
>
> **文件层——safePath 路径穿越校验**：所有路径操作先过 safePath 函数。具体做法是 path.resolve(workDir, filePath) 把 `..` 全解析掉得到真实绝对路径，然后检查这个路径是否以 workDir 开头。如果 AI 传入 `../../etc/passwd`，resolve 后变成 `/etc/passwd`，不以 workDir 开头就直接拒绝，返回"路径不允许"的错误给 AI。这个方案简单但有效，覆盖了所有路径穿越变体（`../`、符号链接等）。
>
> **命令层——run_command 白名单**：只允许 `npm run build`、`npm test`、`npx tsc --noEmit` 等预定义命令前缀，其他一律返回"不允许执行"。为什么用白名单不用黑名单——黑名单永远有遗漏，`rm` 可以写成 `\rm`、`$(rm ...)`、反引号嵌套，AI 的组合能力比人想象的强。白名单是唯一能保证 100% 安全的方案。另外命令有 60 秒超时，防止 AI 执行一个 `while true` 卡住整个进程。
>
> **Git 层——Token 隔离**：Git Token 不暴露给 Agent。Git 操作（clone/commit/push/createPR）是执行引擎的前后置钩子，不是 AI 能调用的工具。AI 在 Agent Loop 中的工具列表里看不到 git 相关工具，也无法执行 `git push --force` 或 `git branch -D` 等危险操作。Token 存在节点配置中，由执行引擎在 AI 完成编码后自动读取并执行 Git 流程。
>
> **代码安全的软性防护**：System Prompt 里告诉 AI 避免 SQL 注入、XSS、硬编码密钥（这是 soft 约束）。Agent 写完代码后用 run_command 跑 `npx tsc --noEmit`（类型检查）+ `npm run lint`，失败了 AI 会自动修复再验证，最多 3 轮自愈。最后一道防线是 Diff 审批——100% 的 AI 产出都经过人工逐文件检查确认。
>
> 一个我设计时考虑但没做的：文件大小限制。如果 AI 写一个 10MB 的文件，当前 write_file 不会拒绝。生产环境应该加单文件上限（如 1MB）和单次执行总写入量限制。

### Q8: Diff 审批怎么实现？

**关键词：** scanDirectory / snapshot / before-after / added-modified-deleted / react-diff-viewer / 回滚

**标准回答：**
1. **背景：** AI 写的代码必须 100% 经过人工确认才能进代码库
2. **实现：** 执行前 scanDirectory 递归扫描目录存为 Map（排除 node_modules/.git/dist，单文件>512KB 不读）→ 执行后再扫描 → 对比得到 added/modified/deleted → 存 execution 记录
3. **前端：** react-diff-viewer 渲染 before/after 对比
4. **驳回回滚：** added 文件 fs.rmSync 删除，modified 用 snapshot 旧内容覆盖写回，deleted 从 snapshot 恢复
5. **取舍：** 为什么不用 Git diff——不是所有节点都配了 Git。改进是两条路并存，有 Git 用 git diff HEAD（更快更标准），没 Git 走快照

**详细口述版：**

> 实现方式是执行前后对比：
>
> **执行前**：scanDirectory 递归扫描工作目录，把每个文件的路径和内容存成一个 Map（`{ 'src/app.ts': '文件内容...' }`），保存到 MySQL 的 snapshot 字段（JSON 格式）。扫描时排除 node_modules、.git、dist 这三个目录——它们占文件量的 99%。单文件超过 512KB 不读内容，只记录路径。
>
> **执行后**：Agent 修改完成后，再次扫描得到 current Map，和 snapshot 对比：current 有而 snapshot 没有 = added；两边都有但内容不同 = modified；snapshot 有而 current 没有 = deleted。生成一个 changes 数组存到 execution 记录中。
>
> **前端展示**：用 react-diff-viewer 渲染 before/after 对比视图，用户能逐文件看到 AI 具体改了什么。
>
> **审批驳回时的回滚**：后端根据 changes 数组逐个还原——added 的文件直接 fs.rmSync 删除；modified 的用 snapshot 中的旧内容覆盖写回；deleted 的从 snapshot 恢复创建。精确到文件级别，不会误伤 Agent 没碰过的文件。
>
> **为什么不用 Git diff**：不是所有节点都配了 Git。有些用户只想让 AI 写几个文件看看效果，不想配仓库。我的改进方案是两条路并存——如果配了 Git 就用 `git diff HEAD`（更快更标准，也更适合大项目），没配 Git 才走快照方案。当前实现只有快照方案。
>
> **性能方面**：排除了三大目录后，一个正常项目几十个源文件，扫描 < 100ms，snapshot JSON 约几百 KB。几千个文件的大项目确实会慢，但这种项目一定会配 Git，走 Git diff 就不存在性能问题。还有一个改进方向是只快照 Agent 实际操作过的文件（从 agent_logs 提取 write_file 的路径列表），而不是扫全目录。

### Q9: 为什么用 SSE 不用 WebSocket？

**关键词：** 单向推送 / POST+JWT / fetch+ReadableStream / 不用 EventSource / HTTP 兼容 / 轮询兜底

**标准回答：**
1. **背景：** 这是单向推送场景（服务端→客户端），不需要客户端通过同一连接回传
2. **为什么不用 WebSocket：** SSE 基于 HTTP，不需要连接升级/心跳/重连逻辑，天然兼容 HTTP 中间件（鉴权、代理），实现成本低
3. **为什么不用原生 EventSource：** EventSource 只支持 GET 且不能带自定义 Header。我需要 POST（触发执行）+ Authorization Header（JWT），所以用 fetch+getReader 手动读流
4. **断连处理：** SSE 断了不影响后端执行（Agent Loop 不依赖前端连接）。做了双通道——SSE 实时推 + agent_logs 表持久化 + 轮询兜底（3s 刷新，5 次无变化停止）
5. **改进：** 如果并发高可以用 Redis Pub/Sub 做多实例间事件广播

**详细口述版：**

> 先说为什么不用 WebSocket：这是单向推送场景——服务端把 Agent 的执行进度推给客户端，不需要客户端通过同一连接回传数据。SSE 基于 HTTP，不需要额外的连接升级、心跳维护、重连逻辑，实现成本低得多。另外 SSE 天然兼容 HTTP 中间件——鉴权（JWT）、代理（Nginx）都不需要单独处理，WebSocket 需要单独配置。
>
> 再说为什么不用浏览器原生 EventSource：因为 EventSource 只支持 GET 且不能带自定义 Header。我的场景需要 POST（触发节点执行时带 body）加 Authorization Header（JWT 鉴权），所以用 fetch + ReadableStream 手动实现 SSE 解析。
>
> 具体实现：后端用 `@Res()` 注入原始 Response 对象，手动写 `text/event-stream` 响应头，然后通过 `res.write('data: ${JSON.stringify(event)}\n\n')` 推送事件。事件类型有 4 种：thinking（轮次开始）、tool_done（工具执行完成，包含工具名、参数、结果）、done（任务完成）、error（执行出错）。
>
> 前端用 `fetch()` 发 POST 请求，拿到 response 后用 `response.body.getReader()` 逐 chunk 读取，按 `\n` 分割后 parse `data:` 开头的行，JSON.parse 得到事件对象，实时更新执行日志的 UI。
>
> 组件间通信用 CustomEvent——ReactFlow 的自定义节点在渲染树很深的位置，和页面级的执行日志组件没有 props 路径。我用 `window.dispatchEvent(new CustomEvent('agent-log-event', { detail }))` 做全局事件总线解耦通信。
>
> **断连处理**：SSE 连接断了不影响后端执行——Agent Loop 跑在服务端，不依赖前端连接，结果照样存数据库。我做了双通道互为备份：实时通道是 SSE 推送，持久化通道是 agent_logs 表（每次工具调用都入库）。用户刷新页面后从 GET API 加载历史日志。另外加了轮询兜底——打开运行 tab 时每 3 秒从 API 刷新日志，连续 5 次无变化（15 秒）才停止。这样即使不通过 SSE 触发的执行，用户也能看到进度。

### Q10: 工作流执行引擎怎么设计？

**关键词：** Kahn 算法 / 入度 / nextMap / 双路径分发 / 审批暂停 / fire-and-forget / 节点间 JSON 协议

**标准回答：**
1. **背景：** 需要按画布连线自动执行多个节点，支持审批暂停
2. **实现：** 计算入度找起点 → 沿 nextMap 顺序执行 → 根据 nodeType 分流（普通走 LLM、代码走 Agent Loop）→ 执行完进入 waiting 等审批 → approve 后 fire-and-forget 继续下游
3. **节点间数据传递：** 结构化 JSON（{ type, text, changedFiles }），下游通过 getNodeInput 读取上游 output → JSON.parse 提取 .text 传给 AI
4. **容错：** 上游旧数据是纯文本 → try JSON.parse catch 后 fallback 返回原文，新旧格式兼容
5. **改进：** 加版本标记——下游记录引用的是上游第 N 次执行的结果，上游更新后前端标黄提示

**详细口述版：**

> 执行引擎的核心在 ExecutionService，负责读取画布数据、确定执行顺序、分发执行、控制审批流程。
>
> **执行顺序的确定**：用简化的 Kahn 算法——遍历 edges 计算每个节点的入度，入度为 0 的就是起点。然后构建 nextMap（从 edges 提取 source → target 的映射），沿 nextMap 顺序执行。当前是线性链（一个节点只有一个后继），所以本质是一个 while 循环沿链走到尾。
>
> **双路径分发**：执行到某个节点时，根据 nodeType 分流。普通节点（PRD 审核、需求分析、Code Review 等）调一次 LLM，传入 System Prompt + 上游输出 + 用户 Prompt，得到一次性文本结果。代码开发节点启动 Agent Loop——进入 while 循环，AI 自主调用工具完成编码任务。
>
> **Human-in-the-Loop 审批**：每个节点执行完后，状态设为 waiting，等待人工审批。前端展示执行结果（普通节点展示 AI 输出文本，代码节点展示 Diff）。用户点 approve 后，后端用 fire-and-forget 模式异步触发下游节点执行——HTTP 立即返回 200，不等下游执行完。这样审批响应 < 100ms。
>
> **节点间数据传递**：每个节点执行完把输出存到 execution.output——是一个 JSON 字符串，格式如 `{ type: 'code_dev', text: 'AI总结', changedFiles: [...] }`。下游执行时通过 getNodeInput 读取上游的 output，JSON.parse 提取 .text 字段（自然语言内容）传给 AI 作为上下文。
>
> **数据一致性**：是快照式的——下游读到的永远是它执行那一刻上游的最新 output。如果用户后来重跑了上游，下游不会自动感知。这是有意的设计：自动级联重跑可能覆盖下游已经审批通过的代码，风险太大，应该由用户决定是否重跑下游。
>
> **容错设计**：getNodeInput 里 try JSON.parse，如果上游是旧数据（纯文本不是 JSON），catch 后直接返回原文。新旧格式的节点可以混用。每个节点的 executeNode 整体在 try/catch 中，失败时 execution.status = 'failed'，工作流停在失败节点，不会继续执行下游。

### Q11: 如果支持 DAG 并行怎么改？

**关键词：** nextMap 一对多 / Promise.all / 汇聚节点 / barrier / 入度>1 / 拓扑排序

**标准回答：**
1. **背景：** 当前 nextMap 是一对一（线性链），不支持并行分支
2. **改造方案：** nextMap 从 Map<string,string> 改为 Map<string,string[]>（一对多）→ 到分支点时 Promise.all 并行执行所有下游 → 到汇聚节点（入度>1）做 barrier 等待所有上游完成 → 全部 approved 后继续
3. **难点：** 汇聚节点的输入如何合并多个上游结果（数组拼接 or 取最后一个）
4. **取舍：** 当前不做是因为线性执行覆盖 90% 场景，并行增加调试复杂度
5. **改进：** 参考 Airflow 的 DAG 执行模型，加节点状态机（pending→running→success/failed）

**详细口述版：**

> 当前实现是线性链——nextMap 是 `Map<string, string>` 的一对一映射，一个节点只记最后一条出边的 target。如果要支持 DAG 并行，核心改动分三步：
>
> **第一步，数据结构改造**：nextMap 从 `Map<string, string>` 改为 `Map<string, string[]>`（一对多）。构建 nextMap 时不再覆盖，而是 push 到数组里。同时维护入度表（inDegree Map）。
>
> **第二步，执行逻辑改造**：到分支点时（nextMap 里对应数组长度 > 1），用 `Promise.all` 并行执行所有下游分支节点。每个分支节点独立执行、独立审批。到汇聚节点时（入度 > 1），做 barrier 等待——检查所有上游节点是否都已完成并 approved，全部通过后才触发汇聚节点执行。
>
> **第三步，汇聚节点的输入合并**：多个上游结果怎么传给汇聚节点？两种策略：一种是数组拼接（把所有上游 output 组成数组传入），一种是按上游类型选择性合并（如代码开发的取 changedFiles，审核的取 verdict）。
>
> **需要加的配套能力**：
> - 节点状态机：pending → running → success/failed → approved/rejected，精确追踪每个并行分支的状态
> - 失败策略：一个分支失败了，其他分支要不要继续？（参考 Airflow 的 trigger_rule：all_success / one_success / all_done）
> - 前端画布：ReactFlow 本身支持多出边，不需要改；但需要加视觉标识显示哪些分支在并行执行
>
> **为什么当前不做**：线性执行覆盖了 90% 的场景——需求分析 → PRD 审核 → 代码开发 → 自测，这个流程天然是串行的。并行场景（如同时开发前端和后端）确实存在，但增加了调试复杂度和状态管理成本。当前优先保证核心链路稳定。

### Q12: Git PR 自动化怎么做？

**关键词：** simple-git / clone / branch / commit / push / createPR / GitHub+GitLab / Token 不暴露

**标准回答：**
1. **背景：** 代码审批通过后自动推到远程仓库并创建 PR，规避手动操作
2. **实现：** 审批通过 → 执行引擎前置钩子 clone 仓库 → Agent 在 workDir 写代码 → 后置钩子 createBranch → commit(变更文件) → push → 调 GitHub/GitLab API createPR
3. **双平台适配：** GitLab 用 Merge Request API（POST /projects/:id/merge_requests），GitHub 用 Pull Request API（POST /repos/:owner/:repo/pulls）
4. **安全：** Git token 存在节点配置中由执行引擎读取，Agent Loop 的工具列表里没有 git 相关工具，AI 无法调用 push/branch -D 等危险操作
5. **改进：** PR 创建后自动触发公司 CI/CD pipeline（SAST 静态安全扫描），形成完整交付闭环

**详细口述版：**

> Git 自动化覆盖了 clone → branch → commit → push → createPR 的完整流程，用 simple-git 这个 npm 包封装底层 git 命令。
>
> **整体流程**：用户在节点配置中填好 Git 仓库地址和 Token。执行引擎在 Agent 编码前作为前置钩子执行 clone（把远程仓库克隆到 workDir）。Agent Loop 在这个 workDir 中自主读写文件完成编码。编码完成后用户审批通过，执行引擎作为后置钩子：createBranch（分支名自动生成，格式如 `ai/feature-xxx-timestamp`）→ git add 变更文件 → git commit → git push → 调用平台 API 创建 PR。
>
> **双平台适配**：GitHub 用 Pull Request API（`POST /repos/:owner/:repo/pulls`，带 title/body/head/base）；GitLab 用 Merge Request API（`POST /projects/:id/merge_requests`，带 source_branch/target_branch/title）。节点配置中让用户选择平台类型，根据类型走不同的 API 调用逻辑。
>
> **安全设计**：Git Token 绝不暴露给 Agent。Token 存在节点配置的 gitToken 字段中（加密存储），只有执行引擎在 clone 和 push 时读取。Agent Loop 的工具列表里没有任何 git 相关工具——AI 看不到 token，也无法执行 `git push --force` 或 `git branch -D` 等危险操作。Git 操作对 AI 来说是完全透明的，它只知道自己在一个目录里写文件，不知道这个目录是 git 仓库。
>
> **分支命名规范**：自动生成，避免手动操作导致的分支命名混乱。格式包含节点类型 + 时间戳，确保唯一性。
>
> **PR 的 title 和 body**：从 Agent Loop 的最终总结中提取。AI 在 Loop 结束时会输出一段任务总结，包含做了什么改动、涉及哪些文件，直接作为 PR 描述。
>
> **和公司 CI/CD 的衔接**：PR 创建后，后续完全交给公司已有的 CI 体系——自动触发构建、单元测试、SAST 静态安全扫描。如果有高危漏洞阻止合并。我的平台负责"从代码到 PR"，公司 CI 负责"从 PR 到合并"，各管各的。

### Q13: 上下文爆炸怎么解决？

**关键词：** 源头截断 / 500 行硬截 / 滑动窗口 / 保留最近 3 轮 / 不能删消息 / tool_call_id 配对

**标准回答：**
1. **背景：** Agent Loop 每轮追加 assistant + N 条 tool result，一个 read_file 可能几千 token，15 轮后 3-5 万 token
2. **源头截断：** read_file 超 500 行只返回前 200 行 + 提示精确读取。System Prompt 约束 AI"小文件全读，大文件用 search_code"
3. **滑动窗口：** messages.length>12 触发压缩。按轮分组，保留最近 3 轮完整，早期 tool result 超 300 字符截断为前 200 字符 + [已截断]
4. **关键约束：** 不能删消息只能截短。OpenAI API 要求 tool_call 和 tool_result 通过 tool_call_id 配对，删了任何一条就报 400
5. **效果：** 跑 20 轮后 messages 条数还是 20+，但总 token 被控制住，早期大段文件内容变为摘要

**详细口述版：**

> 如果不做管理，一定会爆。Agent Loop 每轮往 messages 里追加 1 条 assistant 回复 + N 条 tool result。一个 read_file 的 tool result 可能有几百行代码（几千 token），一个 run_command 的编译错误输出可能 2000+ token。跑 15 轮后 messages 轻松到 3-5 万 token。
>
> 我做了两层控制：
>
> **第一层：源头截断（最有效）**
> - read_file 超过 500 行的文件强制只返回前 200 行 + 提示"文件较大，请用 startLine/endLine 参数精确读取需要的部分"。一个 800 行文件从占 8000 token 降到 2000 token
> - run_command 输出超过 5000 字符只保留末尾 3000 字符——因为编译错误信息通常在输出最后面
> - System Prompt 里写了工具使用策略：小文件全读，大文件先用 search_code 定位再用 startLine/endLine 片段读取。引导 AI 行为减少无谓的大文件读取
>
> **第二层：滑动窗口压缩**
> - 每轮调 AI 前检查 messages.length > 12 就触发压缩
> - 策略是按"轮"分组（1 条 assistant + 后续的 tool messages = 1 轮），保留最近 3 轮完整内容，更早的 tool result 超过 300 字符就截断为前 200 字符 + "[已截断]"
> - 为什么保留最近 3 轮：Agent 的决策主要依赖"刚读了什么、刚写了什么"。更早的操作只需要知道"做过了"就行，不需要完整内容
>
> **关键约束：不能删消息，只能截短 content**
> - OpenAI 的 API 要求 assistant 消息中每个 tool_call 必须有对应的 role='tool' 消息通过 tool_call_id 配对。如果删了任何一条 tool message，后续请求就报 400 错误
> - 所以只能保留消息结构完整性，只截短 content 字段的内容
>
> **和 Claude Code 的对比**：Claude Code 用 LLM 做摘要压缩——调一次 AI 把历史总结为摘要，语义保留度高但每次压缩多一次 API 调用（成本 + 延迟）。我选滑动窗口是因为用的是便宜模型（DeepSeek/GPT-mini），调用频繁但单价低，直接截断比智能摘要更划算。如果用 Claude 200K 窗口，压缩触发频率很低，偶尔触发一次用 LLM 摘要就合理了。
>
> **实际效果**：跑 20 轮后 messages 条数还是 20+（结构完整），但总 token 被控制住——早期轮次的大段文件内容被截断为摘要，整体 token 数稳定在模型窗口限制内。

### Q14: 并发怎么处理？

**关键词：** Node.js async/await / 瓶颈在 AI API / RPM 限流 / 进程内 Map 锁 / Redis 分布式锁

**标准回答：**
1. **背景：** 瓶颈不在服务端计算，在 AI API 调用
2. **Node.js 并发模型：** I/O 全是 async/await，10 个用户同时触发，10 个 fetch 同时 await，互不阻塞。CPU 几乎不消耗（只做 JSON 解析和拼接）
3. **AI API 限流：** 10 并发 × 每秒 1 次 = 600 RPM，容易触发限流。方案：p-queue 控制并发数
4. **并发锁：** 遇到过用户连点两下导致两个 Agent Loop 同时跑在同一 workDir 互相覆盖文件。解决：用 Map 存 workflowId:nodeKey 作为锁 key
5. **生产化：** Map 锁只在单进程有效 → 换 Redis SET NX EX（分布式锁）。释放用 Lua 脚本保证原子性

**详细口述版：**

> 瓶颈不在服务端计算，在 AI API 调用。
>
> **Node.js 并发模型**：虽然 Node.js 是单线程，但 I/O 全是 async/await。10 个用户同时触发 Agent Loop，10 个 fetch 请求同时 await AI API 响应，互不阻塞。CPU 几乎不消耗——服务端只做 JSON 解析和字符串拼接，没有 CPU 密集计算。所以 Node.js 单线程在这个场景下完全不是问题。
>
> **实际瓶颈是 AI API 的 RPM 限制**：每个 Agent Loop 每轮发一次请求，10 个并发 × 每秒 1 次 = 600 RPM，容易触发限流。解决方案是加请求队列做限流——比如用 p-queue 控制全局最大并发数，超限时排队等待而不是直接失败。
>
> **我遇到的真实并发问题**：用户连点两下执行按钮，导致两个 Agent Loop 同时跑在同一个 workDir 里互相覆盖文件。一个在 write_file 写 app.ts，另一个同时在 read_file 读 app.ts——读到的是半写入状态。
>
> **解决方式是加并发锁**：用 Map 存 `workflowId:nodeKey` 作为锁 key。第一次触发时 set(key, true)，第二次触发时检查到 key 已存在，直接抛错"该节点正在执行中"。执行完成后 delete(key) 释放锁。
>
> **Map 锁的局限**：只在当前进程内有效。如果用 PM2 cluster 模式跑 4 个进程，用户请求可能落到不同进程，进程 A 的 Map 里有锁，进程 B 不知道——同一个节点被并发执行两次。
>
> **生产化方案——Redis 分布式锁**：
> - 加锁：`SET agent_lock:{workflowId}:{nodeKey} {processId} NX EX 300`
>   - NX：只有 key 不存在时才设置成功（原子操作，谁先设谁拿到锁）
>   - EX 300：300 秒自动过期（防止进程崩溃后锁永远释放不了）
>   - 值存 processId：释放时验证是不是自己的锁
> - 释放锁用 Lua 脚本保证原子性：`if redis.call('get', key) == processId then redis.call('del', key) end`
> - 为什么不用 Redlock：单节点 Redis 的 SET NX EX 对我的场景够用。Redlock 是多 Redis 节点的共识算法，复杂度高，我不需要那么强的一致性保证
>
> **生产部署方案**：PM2 cluster 模式跑多进程分担连接；AI API 调用加限流队列（Redis 令牌桶/滑动窗口）；长任务（Agent Loop）改为后台任务，HTTP 立即返回，用 SSE 推进度。

> - LLM API 超时没有自动重试 → 对 5xx 和超时做指数退避重试（最多 3 次）

> **第六，前端状态管理**：
> - 组件间用 CustomEvent 通信 → 组件多了会变成 event soup → 应该用 Zustand 做全局状态，SSE 事件写入 store，各组件 subscribe 对应的 slice

### "Agent 死循环怎么办？"

**速答：** 两层防护：硬上限 30 轮自动停止 + run_command 60 秒超时。更好的方案是加外部超时熔断（如整个 Agent Loop 10 分钟上限），以及检测低效循环（连续 3 轮调同一工具同一参数 → 强制终止）。

**详细口述版：**

> 当前有两层防护：第一，Agent Loop 有硬上限 30 轮（while 循环中 iteration < maxIterations），超过自动停止并输出当前进度总结。第二，run_command 有 60 秒超时，防止 AI 执行 while true 或者跑一个很慢的命令卡住进程。
>
> 但这两层有盲区——如果 AI 进入低效循环（比如反复 read_file 同一个文件 20 次），没有外部检测机制，只能等 30 轮跑完自动停止，白白浪费了 token。
>
> 更好的方案是三层：
> 1. 硬上限保留（兜底）
> 2. 外部超时熔断：整个 Agent Loop 设一个总时间上限（如 10 分钟），超时直接终止。用 setTimeout + AbortController 实现
> 3. 低效循环检测：记录最近 N 轮的工具调用，如果连续 3 轮调同一个工具传同一个参数（说明 AI 卡住了），强制终止并返回"任务可能过于复杂，建议拆解"
>
> 另外 System Prompt 里也有约束："如果你发现自己在重复操作，请停下来总结当前进展，而不是继续循环"。但这是 soft 约束，AI 不一定遵守。

### "RAG 不准怎么办？"

**速答：** 加三件事：Rerank 模型二次精排（多 100-200ms 但精度显著提升）、标注测试集评估 Recall@5 和 MRR、审核报告展示引用来源让用户自己判断。

**详细口述版：**

> 分三个方向提升：
>
> **第一，加 Rerank**：当前 top-k 完全靠 embedding 向量相似度排序，但 embedding 是通用的"语义相似"，不一定等于"对审核有用"。加一个 Cross-Encoder reranker（如 bge-reranker-v2），对 top-20 初筛结果做精排，重新按"和审核任务的相关性"排序，取 top-8。代价是每次检索多 100-200ms，但精度提升显著。
>
> **第二，混合检索**：当前只用稠密向量（embedding），漏掉了关键词精确匹配的场景。比如搜索"JWT 认证"，embedding 可能返回"Token 鉴权"（语义相近但没提到 JWT 这个词）。加 BM25 稀疏检索做互补，用 RRF（Reciprocal Rank Fusion）融合两路排序结果。Milvus 2.4+ 原生支持 sparse vector。
>
> **第三，可解释性**：在审核报告中展示引用来源——"基于参考资料 1，相似度 72%"，让用户能判断 AI 的结论是否有据可依。我的 retrievedContext 拼接时已经标注了每条的相似度百分比，只需要在最终输出中要求 AI 标注引用。
>
> 另外还有 Query 改写——用户的输入可能很短或很模糊（如"安全"），先用 LLM 改写为更具体的检索语句（如"接口安全认证方案、数据加密、权限校验"），再去检索。相当于用 AI 理解用户意图后再搜索。

### "Token 爆了怎么办？"

**速答：** 源头截断 + 滑动窗口双重控制。极端情况下压缩保留轮数从 3 动态降到 1。更好方案是 Token 预算制——每轮发送前计算总 token 数，超预算更激进压缩。

**详细口述版：**

> 先说当前方案：两层控制。源头截断——read_file 超 500 行只返回 200 行，从产生端就控制大小。滑动窗口——messages 超 12 条触发压缩，保留最近 3 轮完整，早期 tool result 截短为 200 字符。
>
> 极端情况下（System Prompt 巨长 + 跑了很多轮），压缩的保留轮数可以从 3 动态降到 2 或 1——也就是只保留"当前正在做的这一轮"的完整内容，更早的全部截短。
>
> 更好的方案是 Token 预算制：设一个窗口预算（如模型窗口的 80%，比如 32K 的 80% = 25K token），每轮发送前用 tokenizer（如 tiktoken）计算当前 messages 总 token 数。如果超了预算就更激进地压缩——保留更少轮次、截断更短、甚至对 System Prompt 中的项目结构部分做简化。
>
> 我没做 Token 预算制的原因：需要引入 tiktoken 依赖做精确 token 计算，增加了计算成本（每轮发送前要遍历所有 messages 算 token 数）。当前的 messages.length > 12 是简单近似——不精确但实现零成本。对于 DeepSeek 32K 窗口来说，12 条触发阈值配合截断策略在实际测试中能稳定跑 20 轮不超窗口，够用了。
>
> 还有一个方向是按工具类型差异化压缩：write_file 的结果只有一行（"已创建文件 xxx.ts, 58 行"），根本不需要压缩。read_file 和 run_command 的结果是膨胀主因，应该只对它们做压缩，其他保留完整。

### "为什么不用成熟框架？"

**速答：** Agent Loop 核心代码不到 200 行，直接调 OpenAI SDK。LangChain 的价值在于编排复杂链路，我的场景确定性强、链路固定，包 4-5 层抽象反而增加调试成本。

**详细口述版：**

> 这个问题要分两部分回答——Agent Loop 为什么不用框架，RAG 为什么不用框架。
>
> **Agent Loop 部分**：Tool-Calling 本质就是一个 while 循环 + JSON schema 匹配。OpenAI SDK 原生支持 tool_choice 和 function_calling，直接用 SDK 实现核心代码不到 200 行——一个 while 循环、一个 executeTool 分发函数、5 个工具实现。如果用 LangChain 的 AgentExecutor，它在循环之上包了 Chain / Memory / Callback / OutputParser 等抽象，我用不到但必须理解它的执行流才能 debug。
>
> **RAG 部分**：我的 RAG 链路是确定性的、线性的——解析 → 切片 → Embedding → 入库 → 搜索。每一步对应一个明确的库调用（pdf-parse / 我的 splitText / OpenAI SDK / Milvus SDK），不需要 LangChain 的 Chain 编排。而且我需要精细控制每一步的行为——批量大小 10（阿里云限制）、重试指数退避、Markdown 按标题切片。在 LangChain 中要 override 默认行为，不如自己写 20 行清晰。
>
> **什么时候该用框架**：如果需要快速实验多种检索策略（BM25 + 向量 + 混合 + Rerank），对比不同 Embedding 模型的效果，或者做复杂的多轮对话 + 检索 + 工具调用的编排，LangChain 的统一抽象确实省事——切换 VectorStore 只改一行代码。但我的场景确定性强、不需要频繁切换，自己写更合适。
>
> 核心观点是：框架的价值在于抽象复杂性。如果底层复杂性本身就很低（100 行代码能搞定），框架的抽象反而增加了理解成本和调试难度。
