Agent Loop 怎么实现的？

Agent Loop是一个 while 循环，把 system prompt 和 messages 组合起来发送给模型，如果模型返回 tool_calls , 解析 tool_calls 中的工具和参数，执行这些工具，
然后把这些工具的执行结构作为 role=tool 的消息与messages 合并后在发送给模型，直到模型返回总结性文本的时候才结束循环，这样的循环就是 Agent Loop。
在我们的项目中，我增加了一个优先级，优先使用 codex cli，如果检测到本地未按照 codex cli，自动降级为自研的 Agent Loop。

既然你说了 Agent Loop 是一个while循环，那么多轮执行的时候，肯定会出现死循环的情况，这个你是怎么解决的？

我的解决方案是分为了三种控制：第一是增加循环轮次上限，当执行简单任务的时候，轮次设置为 10到15 轮，当复杂任务的时候 20到30 轮。
第二是同一个工具执行错误了三次，那么可以认为是死循环了，直接停止循环
第三是同一个工具和参数执行了三次以上，也直接停止循环
最后给出用户提示消息，告诉用户在哪轮出现了错误，让用户自己调整 prompt 或者 手动介入。

既然你说了设置最大上限轮次，那如果我的任务正常就会超出 30 轮，你要怎么办？

第一点事 30 轮还做不完，说明节点定义太大，应该在工作流编排时拆细
第二是兜底方案：到达上限后生成总结，告诉用户"任务卡在第 X 步、原因是 XXX"，让用户调整 Prompt 重跑或手动介入——把决策权交给人，不是无限继续
第三是如果非要让 AI 接续：是用户在前端手动续跑这个节点，把上一轮的总结作为新一轮的输入，这是产品交互层的设计，不是循环里偷偷接着跑

那你们肯定碰到过 上下文膨胀的问题，你们是怎么优化的？

第一是源头截断，当执行工具的结果超过 500 行的时候，直接去长度进行限制，只取前 200 行和后200行，中间省略。
第二个是滑动窗口，保留最近几轮的内容，更早的轮次只保留摘要，比如在多少轮，执行了什么工具，返回了什么结果。
第三个事截短内容，因为 tool calling 要求 必须 assistant 中的 tool_call_id 必须与结果的 tool result 一一匹配。如果删除消息，会导致 模型报错 400，所以我们只截短 content 中的内容。
第四个是token监控，当上下文的内容达到限制的 token 的 80%，就进行以上这些操作。

RAG 的实现流程？

我们的RAG分为两条链路
第一是上传链路，当用户上传文档或者输入内容后，把文档解析成纯文本，然后进行chunk，再去调用 embeding 模型进行向量化转换，最后存入 向量库和 数据库
第二是检索链路，当用户输入内容，然后通过 embeding 进行向量化转换，然后在向量库里进行查询相似度比较高的，返回 top-5.

你们的 chunk 切片 是怎么做的？

我们会首先判断文本是否很长，超过 5000 字符，如果超过，就按段落进行分割，如果段落过长就按换行分，如果还是不够，就按 字数分。
并且我们增加了 200 字符的重叠。每片分片大小大约是 512 token 页就是大约 1500 字符左右。

为什么 chunk_size 是 512？

512 是个经验最优值，本质是在语义完整性、检索精度和上下文成本之间做权衡。
太小比如 128，一段话被切碎，召回出来的片段不完整，LLM 拿到也用不了。
太大比如 2048，一个 chunk 里塞了好几个主题，向量是整段的平均语义，会把相关和不相关的内容糅在一起，相似度算出来反而不准。而且 top-K 取 5 的话，单 chunk 2000 token 就是 10000 token，加上 system prompt 和对话历史，窗口很容易撑爆。
512 token 大概对应两三个段落，刚好是一个完整的语义单元。再加上 OpenAI text-embedding 系列、bge 这些主流 embedding 模型本身就在 512 这个量级附近做的优化,超出去效果反而下降。

那 chunk_overlap 200 字符为什么这么定？

overlap 大约是 chunk_size 的 13%,业界经验值一般落在 10% 到 20%。
它解决的核心问题是语义被边界切断,比如一段话刚讲到一半 chunk 边界切下去,前一段是上半句、后一段是下半句,用户 query 命中的关键词正好横跨边界,两个 chunk 召回相似度都不够高,答案就被漏掉了。加 200 字符重叠之后,边界附近的内容会同时出现在前后两个 chunk 里,相当于给关键句一个双保险。
为什么不是 50 或 500:50 太短,中文一句话经常 30 到 80 字,覆盖不了一个完整句子,起不到容错作用;500 又太长,重复内容占比 33%,存储和检索成本都涨,而且会让 top-K 召回出来很多语义雷同的片段,多样性反而下降。
另外这个值跟 chunk_size 是按比例联动的,不是绑死的数字。如果 chunk 改成 512,overlap 一般跟着调到 50 到 100,保持 10% 到 20% 这个比例就行。

你们现在的检索如果出现质量问题，要怎么解决？

第一是增加混合检索，目前是直接通过 embeding 模型去检索的，对于关键词很不友好，所以增加 BM25 去做关键词匹配，最后 通过 RRF 融合两路排序结果，并且 Milvus 原生支持 sparse vector。
第二是增加一个 Cross-Encoder reranker，对粗排出的 top-20 进行一个精排，取 top-5，会对每轮增加 100ms-200ms。
