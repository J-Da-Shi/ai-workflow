## 一、安全与工程化

### 1. 内存文件系统沙盒（memfs）

核心思想：AI 的所有文件操作都在内存中进行，不直接碰真实磁盘。只有通过验证后，才一次性写入。

```
真实磁盘 (Git 仓库)          内存沙盒 (memfs)
  src/                         src/
    auth.service.ts              auth.service.ts (已修改)
    auth.controller.ts           auth.controller.ts (已修改)
                                 captcha.service.ts (新文件)

AI 所有的 write_file → 写入内存沙盒
AI 的 read_file → 优先读内存沙盒，没有则读真实磁盘
验证通过 → 内存沙盒 flush 到真实磁盘 → git add + commit
```

**为什么用 memfs？**

- AI 写错了不会污染真实仓库
- 可以在内存中跑 build 验证（通过 overlay 文件系统）
- Flush 是原子操作，要么全写入要么不写入
- 用户取消执行时，内存直接释放，零副作用

### 2. Task-Level Commit（逻辑单元提交）

不再每写一个文件就 commit。而是 AI 每完成一个逻辑单元（Plan 中的一个步骤），才做一次 commit：

```
Plan:
  Step 1: 创建验证码 Service
  Step 2: 修改登录接口
  Step 3: 更新 DTO

执行过程:
  write_file captcha.service.ts   ─┐
  write_file captcha.module.ts    ─┤→ Commit 1: "feat: add captcha service"
  修改 app.module.ts 注册模块     ─┘

  修改 auth.controller.ts        ─┐→ Commit 2: "feat: integrate captcha into login"

  修改 login.dto.ts              ─┐→ Commit 3: "feat: add captcha field to login DTO"
```

好处：每个 commit 是一个完整的、可理解的改动单元。用户审批时看到的不是零散的文件，而是有业务含义的步骤。

### 3. 并发控制与状态锁

```typescript
// AgentService 中维护运行状态
// key: `${workflowId}:${nodeKey}`
private runningAgents = new Map<string, boolean>();

async runAgent(workflowId, nodeKey, ...) {
  const lockKey = `${workflowId}:${nodeKey}`;

  if (this.runningAgents.get(lockKey)) {
    throw new Error('该节点正在执行中，请等待完成');
  }

  this.runningAgents.set(lockKey, true);
  try {
    // ... Agent Loop
  } finally {
    this.runningAgents.delete(lockKey);
  }
}
```

**前端配合：**

- 执行按钮在 Agent 运行时变为 disabled + "执行中..."
- SSE 断开重连时，先查询 Agent 状态，如果还在运行就继续显示进度

### 4. Git 冲突处理

执行前自动拉取最新代码：

```bash
git fetch origin
git checkout -b ai/task-{nodeKey}-{timestamp} origin/main
```

如果 AI 执行过程中 main 分支被更新（其他人推了代码），在合并阶段提示用户：

```
⚠️  main 分支在 Agent 执行期间有新的提交。
请选择：
  [1] 基于最新 main 重新执行（推荐）
  [2] 强制合并（可能有冲突）
  [3] 保留分支，稍后手动处理
```

### 5. NestJS 规范约束

System Prompt 中注入红线规则：

#### 代码规范（必须遵守）

**1. 依赖注入：**
- 永远使用构造函数注入，禁止 `new Service()`
- 新建的 Service 必须在对应 Module 的 providers 中注册
- 新建的 Controller 必须在对应 Module 的 controllers 中注册
- 新建的 Module 必须在 AppModule 的 imports 中注册

**2. 文件命名：**
- Service: `xxx.service.ts`
- Controller: `xxx.controller.ts`
- Module: `xxx.module.ts`
- Entity: `xxx.entity.ts`（放在 `entities/` 子目录）
- DTO: `xxx.dto.ts`（放在 `dto/` 子目录）

**3. 代码风格：**
- 使用 `@Injectable()`、`@Controller()` 装饰器
- 异步方法返回 `Promise`
- 错误用 NestJS 内置异常（`HttpException`, `NotFoundException` 等）

**Few-Shot 示例**（在 System Prompt 中给标准代码片段）：

```typescript
// 标准的 NestJS Service 写法示例：
@Injectable()
export class XxxService {
  constructor(
    @InjectRepository(XxxEntity)
    private xxxRepo: Repository<XxxEntity>,
    private otherService: OtherService,  // DI 注入
  ) {}

  async findAll(): Promise<XxxEntity[]> {
    return this.xxxRepo.find();
  }
}
```

## 二、Token 优化与成本控制

### 1. 上下文分层管理

```
┌─ 短期记忆（当前 Agent Loop）──────────┐
│  System Prompt (项目结构 + 规范)       │  ← 固定，每次都带
│  Plan (确认后的技术方案)               │  ← 固定
│  最近 5 轮 tool_call + tool_result    │  ← 滚动窗口
│  当前任务描述                         │  ← 固定
└────────────────────────────────────┘

┌─ 历史压缩 ──────────────────────────┐
│  第 6 轮之前的对话 → 压缩为摘要        │
│  "之前读取了 auth.service.ts，发现    │
│   validatePassword 在第 42 行，        │
│   使用了 bcrypt.compare"              │
└────────────────────────────────────┘
```

**具体策略：**

- 保留最近 5 轮完整的 `tool_call` + `tool_result`
- 更早的轮次：`tool_result` 替换为一行摘要（后端调用 AI 做摘要，或用模板生成）
- `read_file` 的结果是最大的 Token 消耗者 — 摘要时只保留"文件名 + 行数 + 关键发现"

### 2. read_file 智能截断

```json
// read_file 工具的参数设计
{
  "name": "read_file",
  "parameters": {
    "path": { "type": "string" },
    "startLine": { "type": "number", "description": "起始行（可选）" },
    "endLine": { "type": "number", "description": "结束行（可选）" }
  }
}
```

**System Prompt 引导 — 文件阅读策略：**

- 小文件（<100行）：直接 `read_file` 全量读取
- 中文件（100-500行）：先 `search_code` 定位关键代码，再 `read_file` 读取相关片段（startLine/endLine）
- 大文件（>500行）：只能通过 `search_code` 检索 + `read_file` 片段读取，禁止全量读取

### 3. 自愈循环的成本控制

| 重试次数 | 行为 |
|---------|------|
| 第1次 | AI 自动修复，正常消耗 |
| 第2次 | AI 自动修复，前端提示"第2次尝试修复..." |
| 第3次 | 失败 → 降级为 Human-in-the-loop |

**降级策略（Human-in-the-loop）：**

```
┌──────────────────────────────────────────┐
│  ⚠️  AI 修复失败（3次尝试均未通过编译）      │
│                                          │
│  错误信息：                               │
│  error TS2345: Argument of type 'string' │
│  is not assignable to parameter of       │
│  type 'number'.                          │
│  → src/auth/auth.service.ts:42           │
│                                          │
│  相关代码：                               │
│  41 │ const result = await this.validate( │
│  42 │   userId,        // ← 这里应该是 number │
│  43 │ );                                  │
│                                          │
│  [🔧 手动修改代码]  [🔄 让AI重试]  [❌ 放弃] │
└──────────────────────────────────────────┘
```

用户可以直接在编辑器里改那一行，改完后点"让 AI 继续"，AI 从修复后的状态继续执行。

### 2. 操作日志的结构化设计

每条 SSE 事件的格式：

```typescript
interface AgentEvent {
  // 事件类型
  type: 'tool_call' | 'tool_result' | 'plan' | 'terminal_output' | 'phase_change' | 'error' | 'complete';

  // 工具调用相关
  toolName?: string;        // read_file, write_file, ...
  toolArgs?: object;        // { path: "src/app.ts" }
  toolResult?: string;      // 执行结果摘要
  toolSuccess?: boolean;    // 是否成功

  // 终端输出
  output?: string;          // 一行终端输出
  isError?: boolean;        // 是否是错误输出（stderr）

  // Plan 相关
  plan?: string;            // Plan 的 Markdown 内容

  // 元信息
  phase?: 'planning' | 'executing' | 'verifying' | 'complete';
  timestamp: string;
  retryCount?: number;      // 当前重试次数（0-3）
}
```

## 未完成功能（按优先级排序）

### P0：Git 集成（让代码真正可交付）

当前 Agent 写的文件只是落在磁盘目录里，没有版本管理。需要：

| 能力 | 说明 |
|------|------|
| clone / pull | 执行前拉取远端仓库到工作目录 |
| 自动创建分支 | `feat/agent-{nodeKey}-{timestamp}` |
| commit | Agent Loop 结束后自动提交 |
| push | 推送到远端 |
| 前端配置 | 节点配置中填写 Git 仓库 URL + 分支策略 |

**实现方式**：作为 Agent Loop 的前后置钩子（不是 AI 调的工具）：
- Loop 前：clone → checkout 新分支
- Loop 后：git add → commit → push

**为什么不让 AI 自己调 git**：避免 AI 乱操作（force push、删分支等），git 操作应该是确定性的框架行为。

---

### P1：Plan 确认流程（Spec-First）

当前 Agent 拿到任务后直接动手写代码。应该先输出 Plan（我要做什么、改哪些文件），等用户确认后再执行。

| 步骤 | 说明 |
|------|------|
| Plan 阶段 | AI 只分析不执行，输出执行计划 |
| 人工确认 | 前端展示 Plan，用户点"确认执行"或"修改" |
| Execute 阶段 | AI 按 Plan 执行文件操作 |

**实现方式**：Agent Loop 分两轮执行：
- 第一轮：System Prompt 指示"只输出计划，不调用 write_file"
- 用户确认后触发第二轮：把 Plan 作为上下文，开始真正执行

---

### P2：run_command 工具 + 自愈循环

让 Agent 能执行 shell 命令（安装依赖、运行测试、编译检查）：

| 能力 | 说明 |
|------|------|
| run_command 工具 | 白名单命令（npm install、npm test、tsc --noEmit 等） |
| 自愈循环 | 执行失败 → AI 读错误 → 修改代码 → 重试（最多 3 次） |

**白名单示例**：`npm install`, `npm test`, `npm run build`, `tsc --noEmit`, `eslint`

---

### P3：沙盒隔离（memfs）

当前直接写真实磁盘，有安全风险。改为内存文件系统：
- Agent 在 memfs 中操作
- 审批通过后 flush 到真实磁盘 / Git

**优先级低**：目前工作目录已经按 workflowId 隔离 + safePath 防穿越，风险可控。等项目要给多人使用时再做。

---

### P4：Diff 审批 + 文件树

审批时不只是看 AI 的文本总结，而是：
- 展示文件树（新增/修改/删除）
- 代码 Diff 查看器（类 GitHub PR 视图）
- 逐文件审批（approve / reject / comment）

# 正在开发中：文件变更 + Diff 审批

## 方案

**当前流程：** Agent 写文件 → 审批（只看 AI 文本总结）→ 通过/驳回

**新流程：** Agent 写文件 → 生成 Diff → 审批时展示代码变更 → 通过（保留文件）/ 驳回（还原文件）

**核心变化：**

- 执行前：对工作目录做快照（记录哪些文件存在、内容 hash）
- 执行后：对比快照和当前状态，生成 diff
- 审批时：前端展示 diff（新增/修改/删除）
- 驳回时：根据快照还原文件

> 不需要沙盒：Agent 还是在正式目录操作，只是我们在执行前后做了"快照 → 对比"。

---

## 逻辑

**执行前：**

1. 扫描工作目录，记录文件列表 + 每个文件内容（快照）
2. 存入 execution 记录

**Agent Loop 执行（不变）：**

3. Agent 正常 read/write/list/search

**执行后：**

4. 再次扫描工作目录
5. 对比快照，生成变更列表：
   - 新增文件：快照中没有，现在有
   - 修改文件：快照中有，现在内容不同
   - 删除文件：快照中有，现在没有（当前工具不支持删除，但预留）
6. 把变更列表存入 execution 记录

**审批通过：**

7. 什么都不做（文件已经在磁盘上了）

**审批驳回：**

8. 根据快照还原：
   - 新增的文件 → 删除
   - 修改的文件 → 用快照内容覆盖还原
   - 删除的文件 → 从快照恢复

# 当前架构与逻辑 — 代码开发节点完整回顾

> 一个能自主写代码的 AI Agent：接收任务描述 → 自动读写文件 → 生成代码 → 展示 Diff → 审批后提交到 Git。

---

## 核心架构：Agent Loop + Tool Use

```
用户给任务 → AI 思考要做什么
  ↓
AI 返回 tool_calls（要调用哪些工具）
  ↓
后端执行工具（读文件/写文件/搜索等）
  ↓
把工具结果反馈给 AI
  ↓
AI 继续思考 → 再调工具 / 或输出最终回复（循环结束）
```

这就是 Agent Loop — 和 Cursor、Claude Code 底层原理一样。

## 各功能模块是怎么做的

### 1. Agent Loop（agent.service.ts — runAgent）

```typescript
for (let i = 0; i < MAX_ITERATIONS; i++) {
  // 1. 发消息给 AI（带工具定义）
  const response = await openai.chat.completions.create({
    messages,    // 对话历史
    tools,       // 可用工具列表
    tool_choice: 'auto',  // AI 自己决定是否调工具
  });

  // 2. AI 没调工具 → 循环结束，返回最终回复
  if (!message.tool_calls) return reply;

  // 3. AI 调了工具 → 逐个执行
  for (const toolCall of message.tool_calls) {
    const result = executeTool(name, args, workDir);
    messages.push({ role: 'tool', content: result });
  }
  // 回到循环顶部，带着工具结果再问 AI
}
```

**关键点：**

- `messages` 数组是对话记忆，每轮追加 AI 回复 + 工具结果
- AI 返回 `tool_calls` 就继续循环，不返回就结束
- 工具执行结果通过 `role: 'tool'` 消息反馈给 AI

### 2. 四个文件工具（agent.tools.ts）

| 工具 | 功能 | 关键实现 |
|------|------|----------|
| `list_directory` | 列出目录结构 | `fs.readdirSync` 递归，最深 5 层 |
| `read_file` | 读取文件 | `fs.readFileSync` + startLine/endLine 行范围 |
| `search_code` | 搜索代码 | `grep -rn` 命令 |
| `write_file` | 写入文件 | `fs.writeFileSync` + 自动创建父目录 |

**安全机制：** `safePath()` 函数校验所有路径不能逃出工作目录（防路径穿越攻击）。

**源头截断：** `read_file` 超过 500 行的文件强制只返回前 200 行 + 提示。

### 3. 上下文管理（agent.service.ts — compressMessages）

**问题：** 跑 15 轮后 messages 膨胀到几万 token，AI 会"失忆"。

**方案：滑动窗口压缩**

```
messages 结构：
  [0] system prompt     ← 永远保留
  [1] user task         ← 永远保留
  [早期轮次]            ← tool result 截断为前 200 字符
  [最近 3 轮]           ← 完整保留
```

- **触发条件：** `messages.length > 12` 时执行压缩
- **约束：** 不能删消息（API 要求 tool_call 和 tool result 配对），只能截短 content

### 5. Diff 审批

- **执行前：** `scanDirectory(workDir)` 扫描所有文件内容作为快照，存入 `execution.snapshot`
- **执行后：** 再次扫描，对比快照生成 changes（added / modified / deleted）
- **审批通过：** 文件已在磁盘上，什么都不做
- **审批驳回：** 根据 snapshot 还原（新增的删除、修改的还原、删除的恢复）
- **前端：** GET `/diff` 接口拿到 before/after，用 `react-diff-viewer-continued` 渲染

### 6. Git 自动化

**执行前（prepareGitRepo）：**

```
有 .git?
  没有 → clone 远端仓库（token 拼入 URL 认证）
  有   → fetch + checkout 基准分支 + pull
→ 创建特性分支 feat/agent-{nodeKey}-{timestamp}
```

**执行后（gitCommit）：**

```
git add . → git commit（本地提交，不 push）
```

**审批通过（gitPushAndCreatePR）：**

```
git push → 调 GitHub/GitLab API 创建 PR/MR → 返回 PR 链接
```

**审批驳回：**

```
checkout 基准分支 → 删除本地特性分支
```

**一句话概括：**

> 我实现了一个类似 Cursor 的 AI 代码开发 Agent，核心是 Agent Loop + Tool Use 架构，支持自主读写文件、上下文压缩防失忆、SSE 实时推送执行进度、Diff 审批确保代码质量、审批通过后自动 Git commit + 创建 PR。

**追问准备：**

| 问题 | 答案要点 |
|------|----------|
| 和 Cursor 的区别？ | 底层原理相同，定位不同：Cursor 是人坐着交互，我的是工作流自动化、无人值守 |
| 为什么不直接用 Claude Code？ | 学习价值 + 平台灵活性（可切模型）+ 成本控制（DeepSeek 便宜） |
| 上下文压缩为什么不用 LLM 摘要？ | 零额外 API 调用、零延迟、成本和效果的平衡点 |
| 为什么用快照+还原而不是沙盒？ | 更轻量，不需要额外目录/内存文件系统，快照天然能生成 Diff |
| Git 为什么在审批后才 push？ | 确保推到远端的代码都是人工审核过的，驳回零成本 |
