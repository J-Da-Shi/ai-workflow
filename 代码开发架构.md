## 一、安全与工程化

### 1. 内存文件系统沙盒（memfs）

核心思想：AI 的所有文件操作都在内存中进行，不直接碰真实磁盘。只有通过验证后，才一次性写入。

```
真实磁盘 (Git 仓库)          内存沙盒 (memfs)
  src/                         src/
    auth.service.ts              auth.service.ts (已修改)
    auth.controller.ts           auth.controller.ts (已修改)
                                 captcha.service.ts (新文件)

AI 所有的 write_file → 写入内存沙盒
AI 的 read_file → 优先读内存沙盒，没有则读真实磁盘
验证通过 → 内存沙盒 flush 到真实磁盘 → git add + commit
```

**为什么用 memfs？**

- AI 写错了不会污染真实仓库
- 可以在内存中跑 build 验证（通过 overlay 文件系统）
- Flush 是原子操作，要么全写入要么不写入
- 用户取消执行时，内存直接释放，零副作用

### 2. Task-Level Commit（逻辑单元提交）

不再每写一个文件就 commit。而是 AI 每完成一个逻辑单元（Plan 中的一个步骤），才做一次 commit：

```
Plan:
  Step 1: 创建验证码 Service
  Step 2: 修改登录接口
  Step 3: 更新 DTO

执行过程:
  write_file captcha.service.ts   ─┐
  write_file captcha.module.ts    ─┤→ Commit 1: "feat: add captcha service"
  修改 app.module.ts 注册模块     ─┘

  修改 auth.controller.ts        ─┐→ Commit 2: "feat: integrate captcha into login"

  修改 login.dto.ts              ─┐→ Commit 3: "feat: add captcha field to login DTO"
```

好处：每个 commit 是一个完整的、可理解的改动单元。用户审批时看到的不是零散的文件，而是有业务含义的步骤。

### 3. 并发控制与状态锁

```typescript
// AgentService 中维护运行状态
// key: `${workflowId}:${nodeKey}`
private runningAgents = new Map<string, boolean>();

async runAgent(workflowId, nodeKey, ...) {
  const lockKey = `${workflowId}:${nodeKey}`;

  if (this.runningAgents.get(lockKey)) {
    throw new Error('该节点正在执行中，请等待完成');
  }

  this.runningAgents.set(lockKey, true);
  try {
    // ... Agent Loop
  } finally {
    this.runningAgents.delete(lockKey);
  }
}
```

**前端配合：**

- 执行按钮在 Agent 运行时变为 disabled + "执行中..."
- SSE 断开重连时，先查询 Agent 状态，如果还在运行就继续显示进度

### 4. Git 冲突处理

执行前自动拉取最新代码：

```bash
git fetch origin
git checkout -b ai/task-{nodeKey}-{timestamp} origin/main
```

如果 AI 执行过程中 main 分支被更新（其他人推了代码），在合并阶段提示用户：

```
⚠️  main 分支在 Agent 执行期间有新的提交。
请选择：
  [1] 基于最新 main 重新执行（推荐）
  [2] 强制合并（可能有冲突）
  [3] 保留分支，稍后手动处理
```

### 5. NestJS 规范约束

System Prompt 中注入红线规则：

#### 代码规范（必须遵守）

**1. 依赖注入：**
- 永远使用构造函数注入，禁止 `new Service()`
- 新建的 Service 必须在对应 Module 的 providers 中注册
- 新建的 Controller 必须在对应 Module 的 controllers 中注册
- 新建的 Module 必须在 AppModule 的 imports 中注册

**2. 文件命名：**
- Service: `xxx.service.ts`
- Controller: `xxx.controller.ts`
- Module: `xxx.module.ts`
- Entity: `xxx.entity.ts`（放在 `entities/` 子目录）
- DTO: `xxx.dto.ts`（放在 `dto/` 子目录）

**3. 代码风格：**
- 使用 `@Injectable()`、`@Controller()` 装饰器
- 异步方法返回 `Promise`
- 错误用 NestJS 内置异常（`HttpException`, `NotFoundException` 等）

**Few-Shot 示例**（在 System Prompt 中给标准代码片段）：

```typescript
// 标准的 NestJS Service 写法示例：
@Injectable()
export class XxxService {
  constructor(
    @InjectRepository(XxxEntity)
    private xxxRepo: Repository<XxxEntity>,
    private otherService: OtherService,  // DI 注入
  ) {}

  async findAll(): Promise<XxxEntity[]> {
    return this.xxxRepo.find();
  }
}
```

## 二、Token 优化与成本控制

### 1. 上下文分层管理

```
┌─ 短期记忆（当前 Agent Loop）──────────┐
│  System Prompt (项目结构 + 规范)       │  ← 固定，每次都带
│  Plan (确认后的技术方案)               │  ← 固定
│  最近 5 轮 tool_call + tool_result    │  ← 滚动窗口
│  当前任务描述                         │  ← 固定
└────────────────────────────────────┘

┌─ 历史压缩 ──────────────────────────┐
│  第 6 轮之前的对话 → 压缩为摘要        │
│  "之前读取了 auth.service.ts，发现    │
│   validatePassword 在第 42 行，        │
│   使用了 bcrypt.compare"              │
└────────────────────────────────────┘
```

**具体策略：**

- 保留最近 5 轮完整的 `tool_call` + `tool_result`
- 更早的轮次：`tool_result` 替换为一行摘要（后端调用 AI 做摘要，或用模板生成）
- `read_file` 的结果是最大的 Token 消耗者 — 摘要时只保留"文件名 + 行数 + 关键发现"

### 2. read_file 智能截断

```json
// read_file 工具的参数设计
{
  "name": "read_file",
  "parameters": {
    "path": { "type": "string" },
    "startLine": { "type": "number", "description": "起始行（可选）" },
    "endLine": { "type": "number", "description": "结束行（可选）" }
  }
}
```

**System Prompt 引导 — 文件阅读策略：**

- 小文件（<100行）：直接 `read_file` 全量读取
- 中文件（100-500行）：先 `search_code` 定位关键代码，再 `read_file` 读取相关片段（startLine/endLine）
- 大文件（>500行）：只能通过 `search_code` 检索 + `read_file` 片段读取，禁止全量读取

### 3. 自愈循环的成本控制

| 重试次数 | 行为 |
|---------|------|
| 第1次 | AI 自动修复，正常消耗 |
| 第2次 | AI 自动修复，前端提示"第2次尝试修复..." |
| 第3次 | 失败 → 降级为 Human-in-the-loop |

**降级策略（Human-in-the-loop）：**

```
┌──────────────────────────────────────────┐
│  ⚠️  AI 修复失败（3次尝试均未通过编译）      │
│                                          │
│  错误信息：                               │
│  error TS2345: Argument of type 'string' │
│  is not assignable to parameter of       │
│  type 'number'.                          │
│  → src/auth/auth.service.ts:42           │
│                                          │
│  相关代码：                               │
│  41 │ const result = await this.validate( │
│  42 │   userId,        // ← 这里应该是 number │
│  43 │ );                                  │
│                                          │
│  [🔧 手动修改代码]  [🔄 让AI重试]  [❌ 放弃] │
└──────────────────────────────────────────┘
```

用户可以直接在编辑器里改那一行，改完后点"让 AI 继续"，AI 从修复后的状态继续执行。

## 三、用户体验（UX）

### 1. SSE 实时推送 + 流式终端

Agent 执行过程中，前端通过 SSE 实时接收每一步：

```
┌─────────────────────────────────────────────────┐
│  Agent 执行面板                                   │
├─────────────────────────────────────────────────┤
│                                                 │
│  📋 Phase: Planning                              │
│  ✅ list_directory src/modules/auth/  (3 files)  │
│  ✅ search_code "validatePassword"               │
│     → auth.service.ts:42                         │
│  ✅ read_file src/modules/auth/auth.service.ts   │
│  📝 Plan 已生成，等待确认...                      │
│                                                 │
│  ─── 用户确认方案 ───                             │
│                                                 │
│  🔨 Phase: Execution                             │
│  ✅ write_file src/common/captcha.service.ts     │
│  ✅ write_file src/common/captcha.module.ts      │
│  ✅ write_file src/modules/auth/                 │
│     auth.controller.ts (modified)                │
│  🔄 run_command npm run build                    │
│                                                 │
│  ┌─ 终端输出 ──────────────────────────────────┐ │
│  │ > tsc --noEmit                              │ │
│  │ src/auth/auth.controller.ts(15,3):          │ │
│  │   error TS2345: Argument of type...  ← 红色  │ │
│  └─────────────────────────────────────────────┘ │
│                                                 │
│  🔄 AI 正在修复错误...                            │
│  ✅ write_file src/modules/auth/                 │
│     auth.controller.ts (fix)                     │
│  ✅ run_command npm run build                    │
│                                                 │
│  ┌─ 终端输出 ──────────────────────────────────┐ │
│  │ > tsc --noEmit                              │ │
│  │ Build completed successfully.        ← 绿色  │ │
│  └─────────────────────────────────────────────┘ │
│                                                 │
│  ✅ Agent 执行完成，等待审批                       │
└─────────────────────────────────────────────────┘
```

**终端输出的处理：**

- `run_command` 执行时，后端通过 `child_process.spawn` 拿到 stdout/stderr 流
- 逐行通过 SSE 推送给前端
- 前端用正则匹配 `error`/`Error`/`ERROR` 关键字，渲染为红色
- `success`/`completed`/`passed` 渲染为绿色

### 2. 操作日志的结构化设计

每条 SSE 事件的格式：

```typescript
interface AgentEvent {
  // 事件类型
  type: 'tool_call' | 'tool_result' | 'plan' | 'terminal_output' | 'phase_change' | 'error' | 'complete';

  // 工具调用相关
  toolName?: string;        // read_file, write_file, ...
  toolArgs?: object;        // { path: "src/app.ts" }
  toolResult?: string;      // 执行结果摘要
  toolSuccess?: boolean;    // 是否成功

  // 终端输出
  output?: string;          // 一行终端输出
  isError?: boolean;        // 是否是错误输出（stderr）

  // Plan 相关
  plan?: string;            // Plan 的 Markdown 内容

  // 元信息
  phase?: 'planning' | 'executing' | 'verifying' | 'complete';
  timestamp: string;
  retryCount?: number;      // 当前重试次数（0-3）
}
```

## 六、分期实施（终版）

### Phase 1 — Agent Loop + 文件工具 + Plan 确认

- **后端：** AgentModule（service + tools + controller）
- 4 个文件工具（read_file, write_file, list_directory, search_code）

### Phase 3 — 闭环验证 + Diff 审批

- run_command 工具（白名单）+ 自愈循环（3次上限）
- Human-in-the-loop 降级
- 流式终端输出 + 错误高亮
- Diff 查看器 + 逐步骤审批 + 文件树
