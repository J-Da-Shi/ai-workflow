## 2. AI Agent 工作流平台

### 问题：这个 AI 工作流平台和普通 AI Chat 有什么区别？

**口述答案：**

普通 AI Chat 主要是用户问、模型答，结果通常是一段文本。而这个平台是面向研发流程的执行系统，重点不是“聊”，而是“执行”和“交付”。

它有三个核心区别：

第一，有工作流编排。PRD 审核、代码开发、测试、Review、上线审批都可以抽象成节点，节点之间有状态流转和依赖关系。

第二，有 Agent 工具调用。模型不是一次性输出代码，而是可以多轮调用工具，先读项目、搜索代码、写文件、执行命令，再根据结果继续修复。

第三，有人工审批闭环。AI 生成代码后，系统会生成 Diff，用户审批通过后才会进入 Git 仓库，驳回可以回滚。

所以它更像是一个 AI 研发流程自动化平台，而不是普通聊天机器人。

### 问题：一次代码开发节点的完整执行链路是什么？

**口述答案：**

完整链路我会分成七步：

1. 用户在 ReactFlow 画布中选择代码开发节点，配置 Prompt、Git 仓库、知识库和执行引擎。
2. 点击执行后，前端调用后端的流式执行接口，用 `fetch + ReadableStream` 接收 SSE。
3. 后端创建 execution 记录，把节点状态改成 running。如果配置了 Git，会先 clone 仓库并创建特性分支。
4. 后端启动 Agent Loop，把任务、工具 schema、工作目录和上下文发给模型。
5. 模型返回 tool calls 后，后端执行文件读取、代码搜索、写文件、命令执行等工具，再把 tool result 追加回上下文，继续下一轮。
6. Agent 结束后，后端对比执行前快照和当前文件，生成代码 Diff，把节点状态改为 waiting。
7. 前端展示 Diff，用户审批通过后后端 commit、push 并创建 PR；如果驳回，就根据快照还原文件。

这个链路的核心是：AI 可以执行，但最终交付必须可审查、可回滚、可追踪。

### 问题：你在这个项目里最难的点是什么？

**口述答案：**

我觉得最难的是把 Agent 的不确定执行过程做成稳定、可理解的产品体验。

AI Agent 执行过程中可能会多轮思考、多次调用工具、读写很多文件、执行命令，还可能失败重试。如果前端只显示一个 loading，用户完全不知道系统在做什么，也不敢信任 AI 的产出。

所以我把执行过程拆成可视化事件，比如 thinking、tool_done、done、error，通过 SSE 实时推送到前端。前端用时间线展示工具调用、参数、结果和最终回复。执行完成后，再用 Diff 审批把文件级变更展示出来。

这样用户可以看到 AI 为什么这么改、改了哪些文件、结果是否可靠。

## 3. Agent Loop

### 问题：什么是 Agent Loop？

**口述答案：**

Agent Loop 是一种让模型通过多轮工具调用完成任务的执行机制。它不是让模型一次性输出最终答案，而是给模型一组工具，比如读取文件、搜索代码、写文件、执行命令，让模型根据当前上下文决定下一步调用哪个工具。

一次循环大概是：

```text
模型接收任务和工具列表
-> 模型返回 tool_calls
-> 后端执行工具
-> 工具结果作为 tool message 回填
-> 再次请求模型
-> 直到模型不再调用工具，输出最终答案
```

在代码开发场景里，Agent Loop 可以模拟开发者的工作方式：先理解项目，再定位文件，再修改代码，再验证，再修复。

### 问题：Tool Use 协议怎么设计？

**口述答案：**

Tool Use 协议主要包含三部分：

第一是工具声明，也就是告诉模型有哪些工具，每个工具叫什么、作用是什么、参数 schema 是什么。

第二是工具调用，模型返回 `tool_calls`，其中包含工具名和 JSON 参数。

第三是工具结果，后端执行工具后，把结果作为 `role=tool` 的消息返回给模型，并通过 `tool_call_id` 和前一次调用关联起来。

比如代码开发节点里常见工具包括：

```text
list_directory：列目录
read_file：读取文件
search_code：搜索代码
write_file：写文件
run_command：执行验证命令
```

这个设计的关键是后端掌控工具执行权，模型只负责提出调用意图，不能直接操作系统。

### 问题：为什么 tool_call 和 tool result 必须一一对应？

**口述答案：**

OpenAI 兼容的 Tool Calling 协议要求，每个 assistant 返回的 tool call 后面都必须有对应的 tool result。模型发起了一个工具调用，后端执行后必须用相同的 `tool_call_id` 把结果补回上下文。

如果中间删除了某个 tool result，或者上下文压缩时破坏了这个配对关系，后续模型请求可能会报错，或者模型无法理解工具执行结果。

所以在做上下文压缩时，不能简单删除旧消息，而要保留结构，只压缩 tool result 的内容。

### 问题：Agent 如何避免无限循环？

**口述答案：**

我会从四个层面控制：

第一，设置最大 Agent 轮数，比如 30 轮，超过就强制停止。

第二，设置失败自愈次数，比如编译失败后最多修复 3 轮，超过就交给人工处理。

第三，工具结果要结构化返回，让模型知道失败原因，而不是一直重复同一个无效动作。

第四，任务设计上鼓励拆节点，不让一个节点承担过大的任务。工作流编排本身就是为了把大任务拆成可控步骤。

### 问题：失败自愈最多 3 轮具体怎么做？

**口述答案：**

失败自愈主要发生在验证阶段。Agent 写完代码后，后端会执行类型检查、构建或测试命令。如果命令失败，后端把 stderr、错误文件、错误行号等信息作为工具结果返回给模型。

模型拿到错误信息后继续修改代码，再次执行验证。这个过程最多重复 3 轮。

设置上限是为了防止模型陷入反复修复同一个问题的循环。如果 3 轮后还失败，系统会停止并保留错误日志，让人工介入。

### 问题：Agent 单节点修改 10+ 文件，你怎么证明？

**口述答案：**

在代码开发节点里，一个需求通常会涉及多层文件，比如后端的 controller、service、dto、entity，前端的 api、页面组件、状态类型、样式文件，甚至还会涉及测试或配置。

Agent Loop 的优势是模型可以连续调用工具：先搜索项目结构，再读取相关文件，最后多次调用写文件工具创建或修改多个文件。执行结束后，系统会通过 Diff 展示所有变更文件。

所以“10+ 文件修改”不是一次性生成大段代码，而是多轮工具调用累积出来的真实文件变更。

## 4. RAG 和 Milvus

### 问题：RAG 是什么？

**口述答案：**

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。它的核心思想是：模型回答前，先从外部知识库里检索相关资料，再把这些资料作为上下文提供给模型。

在我的项目里，RAG 用在 PRD 审核。团队规范文档会先被解析、切片、向量化并存入 Milvus。审核 PRD 时，系统会把 PRD 文本转成向量，检索相关规范，再让模型基于这些规范做结构化评分。

这样审核结果不是纯模型主观判断，而是有团队规范作为依据。

### 问题：文档从上传到入库的完整链路是什么？

**口述答案：**

完整链路是：

```text
上传文档
-> 解析文本
-> 递归切片
-> 批量生成 embedding
-> Milvus 存向量和元数据
-> MySQL 存知识库、文档和 chunk 记录
```

MySQL 存业务数据，方便列表展示、权限控制、文档状态管理；Milvus 存向量，负责相似度检索。

删除知识库或文档时，需要同步删除 MySQL 记录和 Milvus 向量，避免出现业务数据没了但向量仍被检索出来的脏数据。

### 问题：为什么要递归切片？

**口述答案：**

切片的难点是平衡语义完整性和检索精度。

如果 chunk 太大，一个片段里会混入很多无关内容，向量相似度可能不够准确，模型拿到上下文后噪声也多。

如果 chunk 太小，又会丢失语义上下文，比如一个规范条款被切碎后，模型不知道它的适用条件。

所以更合理的做法是递归切片：优先按标题、段落、列表分割，超过长度再按句子或固定字符切，并保留 documentType、knowledgeBaseId、chunkIndex 等元数据。

### 问题：Milvus 里存什么？

**口述答案：**

Milvus 里主要存 chunk 级别的数据：

```text
id：chunk id
knowledge_base_id：所属知识库
document_type：文档类型
content：原始切片文本
embedding：向量
```

检索时用 embedding 做向量相似度搜索，用 knowledge_base_id 做过滤，避免不同知识库之间互相干扰。

### 问题：PRD 审核的结构化评分怎么设计？

**口述答案：**

我会把审核结果设计成固定 JSON 结构，比如：

```json
{
  "totalScore": 82,
  "dimensions": [
    {
      "name": "文档完整性",
      "score": 80,
      "reason": "缺少异常流程说明",
      "suggestion": "补充失败场景和兜底逻辑"
    }
  ],
  "risks": [],
  "summary": ""
}
```

这样前端可以稳定渲染评分卡，后端也方便存储和统计。相比自由文本，结构化输出更适合产品化。

## 5. SSE 实时推送

### 问题：为什么用 SSE，而不是 WebSocket？

**口述答案：**

Agent 执行日志主要是服务端向客户端单向推送，比如 thinking、tool_done、done、error。客户端不需要在同一条连接里频繁发消息，所以 SSE 更轻量。

SSE 基于 HTTP，接入成本低，和现有鉴权、网关、日志体系更容易结合。WebSocket 更适合强双向场景，比如多人协同编辑、在线聊天和实时状态广播。

所以当前场景我优先用 SSE。后续如果做多人协同画布，可以再引入 WebSocket。

### 问题：为什么不用原生 EventSource？

**口述答案：**

原生 EventSource 只能发送 GET 请求，而且不能方便地自定义 Header。我们的执行接口需要带 `Authorization: Bearer token`，并且执行节点更符合 POST 语义。

所以前端用 `fetch + ReadableStream + TextDecoder` 手动读流，后端仍然返回 SSE 格式：

```text
data: {"type":"thinking","data":{"iteration":1}}

data: {"type":"tool_done","data":{...}}

data: [DONE]
```

### 问题：SSE 解析有什么坑？

**口述答案：**

最大的问题是半包和粘包。网络层的 chunk 不等于业务层的 SSE event。一个 JSON 可能被拆成两个 chunk，也可能一个 chunk 里包含多个 event。

如果简单对每个 chunk 做 `split('\n')`，遇到半包 JSON 就会解析失败。

生产级方案应该维护一个 buffer，以 `\n\n` 作为 SSE event 边界：

```text
buffer += 当前 chunk
while buffer 中存在完整 event:
  取出 event
  解析 data
  剩余内容继续留在 buffer
```

还要处理断线重连、错误事件、超时和最终日志兜底。

### 问题：SSE 和轮询为什么同时存在？

**口述答案：**

SSE 负责单次执行过程的实时展示，让用户看到 Agent 当前在思考第几轮、调用了什么工具、结果是什么。

轮询负责状态最终一致，尤其是批量工作流执行时，前端不一定一直持有每个节点的 SSE 连接，所以需要定期拉 execution 状态，把画布节点状态同步回来。

简单说，SSE 解决过程可见，轮询解决最终一致。

## 6. ReactFlow 工作流画布

### 问题：为什么用 ReactFlow？

**口述答案：**

这个项目需要可视化编排研发流程，用户要能拖拽节点、连线、查看节点状态、打开节点配置和审批结果。ReactFlow 对节点、边、拖拽、缩放、自定义渲染、状态管理支持比较完整，比自己从零实现 SVG 或 Canvas 成本低很多。

更重要的是，ReactFlow 的节点和边可以承载业务状态。比如节点状态变化后，自定义节点可以变色，自定义边也可以根据源节点和目标节点状态变成绿色、黄色或灰色。

### 问题：节点和边怎么设计？

**口述答案：**

节点 data 会存业务信息，比如：

```text
key：节点业务标识
name：节点名称
status：执行状态
summary：执行摘要
workflowId：所属工作流
color/icon：展示信息
```

边表示节点之间的执行依赖。自定义边会读取源节点和目标节点状态，如果两端都 approved，就显示绿色实线；如果目标节点 running 或 waiting，就显示黄色虚线；其他情况显示灰色。

这样用户不用打开详情，也能通过画布直观看到流程进行到哪一步。

### 问题：节点状态怎么同步？

**口述答案：**

节点状态以后端 execution 为准，前端做两层同步。

第一，进入画布时读取 canvas 和 executions，把执行状态合并到节点 data。

第二，执行中通过 SSE 更新当前节点日志，同时通过轮询同步所有节点 execution 状态。

审批操作会做乐观更新，比如点击通过后先把当前节点显示成 approved，同时派发同步事件，最终以后端返回状态为准。如果接口失败，就回滚状态。

### 问题：画布性能怎么优化？

**口述答案：**

我会从几个方面优化：

1. 节点组件 memo 化，避免无关状态导致所有节点重渲染。
2. 拖拽时只更新本地状态，拖拽结束或 debounce 后再保存后端。
3. 大日志、大 Diff 不放进节点 data，只在抽屉打开时懒加载。
4. 节点很多时做分组、折叠、搜索定位和虚拟化。
5. 对画布保存接口做节流，避免高频请求。

## 7. Diff 审批和 Git 自动化

### 问题：Diff 是怎么生成的？

**口述答案：**

代码开发节点执行前，后端会对工作目录做一次文件快照。Agent 执行完成后，再扫描当前工作目录，与快照对比。

对比结果分三类：

```text
added：执行前没有，执行后新增
modified：执行前后都存在，但内容不同
deleted：执行前存在，执行后不存在
```

后端把这些变更存到 execution 里，前端通过 Diff Viewer 展示 before 和 after。

### 问题：为什么审批通过后才 push？

**口述答案：**

因为 AI 生成代码存在不确定性，不能让它直接影响远端仓库。

执行阶段只在本地工作目录和本地分支里修改代码，并生成本地 commit。用户在前端看完 Diff，确认没问题后，后端才 push 分支并创建 PR。

这保证了 AI 代码进入仓库前必须经过人工确认，也方便代码审查和回滚。

### 问题：驳回如何还原？

**口述答案：**

驳回时后端根据执行前保存的 snapshot 和 changes 来还原。

新增文件直接删除；修改文件用 snapshot 中的旧内容覆盖；删除文件则根据 snapshot 恢复回来。最后如果创建了本地分支，也会清理本地分支。

这样用户可以放心让 Agent 尝试修改代码，因为不满意可以一键回退。

### 问题：GitHub 和 GitLab 创建 PR/MR 有什么区别？

**口述答案：**

底层 Git 操作是一样的，都是 clone、checkout branch、add、commit、push。差异主要在平台 API。

GitHub 创建 Pull Request 通常需要 owner、repo、head、base、title、body。

GitLab 创建 Merge Request 通常需要 project id、source_branch、target_branch、title。project id 可能要从仓库路径 URL encode 得到。

所以实现上可以把本地 Git 流程复用，把平台差异封装在 createPR 这一层。

## 8. 安全边界

### 问题：Agent 能执行命令，怎么保证安全？

**口述答案：**

我会从四层控制：

第一，工作目录隔离。Agent 只能在指定 workDir 下操作文件。

第二，路径穿越校验。所有文件路径都 resolve 成绝对路径，再判断是否仍在 workDir 下，防止访问 `../../.ssh/id_rsa`。

第三，命令白名单。模型不能执行任意 shell，只允许有限命令，比如类型检查、测试、安装依赖等。危险命令如 `rm -rf`、修改系统目录、`curl | sh` 都拒绝。

第四，人工审批。Agent 写完代码后，必须经过 Diff 审批，才能进入 Git 远端。

### 问题：RAG 文档里如果有恶意指令怎么办？

**口述答案：**

这是 RAG Prompt Injection 问题。知识库文档虽然是检索内容，但不能被当成系统指令执行。

我会在 system prompt 里明确约束：检索内容只作为参考资料，不得覆盖系统规则。并且把 RAG 内容放在明确的引用区域，要求模型只基于其进行评分和引用。

更进一步可以做文档来源白名单、敏感指令检测、引用溯源和人工审核。

### 问题：Git token 怎么保护？

**口述答案：**

生产环境不应该让 token 明文长期暴露在前端。

我会做几件事：

1. token 后端加密存储，前端不回显明文。
2. 日志和错误信息中脱敏。
3. 使用最小权限 token，只授权指定仓库。
4. 支持用户级凭证隔离。
5. 更理想的是接 GitHub App、GitLab OAuth，而不是用户手动填长期 token。

## 9. 上下文管理

### 问题：为什么 Agent 上下文会膨胀？

**口述答案：**

Agent 每一轮都会产生 assistant 消息和 tool result。特别是 read_file、search_code、run_command 这些工具，返回内容可能非常长。如果几十轮都完整保留，很快就会超过模型上下文窗口。

上下文膨胀的主要来源是大文件内容、搜索结果、命令输出、错误堆栈和多轮工具调用历史。

### 问题：滑动窗口压缩怎么做？

**口述答案：**

核心思路是最近几轮完整保留，早期轮次压缩内容。

最近几轮最影响当前决策，所以保留完整 tool result。更早的轮次只保留关键摘要，比如做过什么、涉及哪些文件、结论是什么。工具返回时也做源头截断，比如限制 read_file、search_code、run_command 的最大输出长度。

但压缩时不能破坏 tool calling 消息结构。assistant 的 tool_calls 和 tool result 必须配对，所以不能直接删除某个 tool result，只能压缩 content。

### 问题：压缩后丢失信息怎么办？

**口述答案：**

压缩本身确实可能丢信息，所以我把它设计成“保留路线图，而不是保留所有细节”。

早期轮次保留文件路径、操作摘要、关键结论。如果模型后续需要完整内容，可以重新调用 read_file 获取。这样既控制 token，又不影响模型按需取数。

## 10. Electron 大模型训练平台

### 问题：Electron 主进程和渲染进程分别做什么？

**口述答案：**

主进程负责应用生命周期、窗口管理、系统能力调用、文件系统访问、进程调度等。渲染进程本质上是前端页面，负责 UI 展示、交互、表单、任务状态和可视化。

在大模型训练平台里，渲染进程负责模型转换任务的配置、任务进度展示、日志展示和结果管理；主进程负责和本地系统能力交互，比如选择文件、启动本地任务、读取日志等。

主进程和渲染进程之间通过 IPC 通信。

### 问题：Electron 怎么做安全隔离？

**口述答案：**

我会关闭渲染进程的 `nodeIntegration`，开启 `contextIsolation`，通过 preload 和 `contextBridge` 暴露有限 API 给前端。

这样前端页面不能直接访问 Node.js 能力，只能调用我们显式暴露的方法，比如选择文件、读取任务状态、打开目录等。

同时 IPC 要做参数校验，避免渲染进程传入危险路径或命令。

### 问题：Monorepo 在 Electron 项目里解决了什么问题？

**口述答案：**

Electron 项目通常有主进程、渲染进程、公共类型、工具函数、组件库等多个模块。如果放在一个包里会越来越耦合。

用 pnpm workspace 做 Monorepo 后，可以把主进程、React 渲染层、共享类型、公共工具拆成独立 package，各自构建和测试，同时复用依赖和类型定义。

这样能提升可维护性、模块复用性和构建效率。

### 问题：你简历里写 ONNX、MNN、TensorRT，你前端具体负责什么？

**口述答案：**

这些是模型工程链路里的格式和推理框架。我的职责不是做底层算法优化，而是负责把这条链路平台化、可视化和工程化。

具体包括任务配置页面、参数校验、模型文件选择、转换任务提交、进度展示、日志展示、失败重试、结果产物管理，以及和后端/算法服务定义接口。

我理解这些模型格式在流程中的位置，但不会把自己包装成算法工程师。

## 11. 微前端与 iframe 沙箱

### 问题：为什么用 iframe 微前端？

**口述答案：**

当时的背景是历史系统多、技术栈不统一、改造成本敏感。iframe 的优势是天然 JS 和 CSS 隔离，接入成本低，对老系统侵入小。

相比 qiankun 或 single-spa，iframe 的缺点是通信、路由同步和性能体验需要额外处理；但在历史系统集成和隔离要求强的场景下，它是一个务实方案。

### 问题：iframe 父子应用怎么通信？

**口述答案：**

主要通过 `postMessage`。父应用向子应用发送用户信息、路由参数、权限信息，子应用也可以向父应用发送页面标题、操作事件、跳转请求等。

安全上要校验 origin，不能接收任意来源消息；消息结构也要定义 type 和 payload，避免随意传对象。

### 问题：iframe 登录态和权限怎么共享？

**口述答案：**

父应用统一登录，子应用启动时由父应用通过 URL 参数或 postMessage 下发 token 和用户信息。

权限上，父应用控制菜单和路由入口，子应用内部根据权限数据控制按钮级和页面级渲染。

如果安全要求更高，可以让子应用通过后端换取短期 token，避免长期 token 在前端多处传递。

## 12. 分片上传与断点续传

### 问题：GB 级文件上传怎么设计？

**口述答案：**

完整流程是：

1. 前端选择文件后，根据文件内容计算唯一 hash。
2. 按固定 chunk size 切片。
3. 请求后端检查已上传分片，支持秒传和断点续传。
4. 前端用并发队列上传未完成分片。
5. 所有分片上传完成后，通知后端合并。
6. 前端展示整体进度、失败重试、暂停和恢复。

核心点是不要一次性上传整个大文件，而是拆成可重试、可恢复的小块。

### 问题：chunk size 怎么选？

**口述答案：**

chunk size 要平衡请求数量和失败成本。太小会导致请求数量过多，服务端压力大；太大则单片失败重传成本高。

一般可以从 2MB、5MB、10MB 这类大小开始，根据网络环境和服务端限制调整。也可以对大文件动态增大 chunk size。

### 问题：大文件 hash 计算会卡主线程怎么办？

**口述答案：**

大文件 hash 计算如果在主线程做，会阻塞 UI。优化方式是用 Web Worker 在后台计算，并且分片增量计算 hash。

前端主线程只负责进度展示和用户交互，Worker 计算完后把 hash 发回主线程。

## 13. 组件库与工程化

### 问题：为什么要做内部组件库？

**口述答案：**

Ant Design 能解决通用 UI，但企业内部会有很多高频业务场景，比如动态表单、权限按钮、上传、可拖拽弹窗、业务筛选器等。如果每个项目重复实现，成本高且体验不统一。

内部组件库的价值是沉淀通用能力，统一交互和代码规范，减少重复开发。

### 问题：动态表单怎么设计？

**口述答案：**

动态表单核心是 schema 驱动。表单项不是写死在 JSX 里，而是由配置描述字段类型、label、默认值、校验规则、联动关系、展示条件和布局。

渲染层根据 schema 选择对应组件，比如 Input、Select、DatePicker、Upload。校验层根据 rules 执行同步或异步校验。复杂场景下还要支持字段联动、嵌套表单、数组表单和拖拽排序。

这样业务方只需要配置 schema，就能生成复杂表单。

### 问题：Monorepo 和 pnpm workspace 有什么价值？

**口述答案：**

Monorepo 适合多个包协同开发，比如前端应用、后端、shared 类型、组件库、工具包。它的价值是统一依赖、统一规范、方便跨包修改和联调。

pnpm workspace 的优势是依赖安装快、磁盘占用低，并且通过软链接组织本地包。相比多个独立仓库，Monorepo 更适合强关联项目。

## 14. 性能优化

### 问题：你做过哪些前端性能优化？

**口述答案：**

我做过几类：

第一，首屏优化：路由懒加载、组件按需加载、资源压缩、减少首屏接口、骨架屏。

第二，渲染优化：虚拟滚动、批量更新、组件 memo、减少无意义重渲染。

第三，大文件和大日志优化：分片上传、Web Worker 计算 hash、日志分页、大 Diff 懒渲染。

第四，构建优化：Vite/Webpack 拆包、依赖分析、Tree Shaking、缓存和 sourcemap 策略。

### 问题：虚拟滚动原理是什么？

**口述答案：**

虚拟滚动的核心是只渲染可视区域附近的列表项，而不是一次性渲染全部数据。

实现上会根据容器高度、滚动位置、单项高度计算当前应该展示的 startIndex 和 endIndex，同时用一个占位容器撑起总高度，再通过 transform 或 padding 把可视项移动到正确位置。

它适合大列表、日志流、表格和 Diff 文件列表。

### 问题：React 重渲染怎么排查？

**口述答案：**

我会先用 React DevTools Profiler 看哪些组件频繁渲染、耗时高。然后检查 props 是否每次都是新引用，比如对象、数组、函数是否需要 memo。

常见优化包括拆分组件、React.memo、useMemo、useCallback、局部状态下沉、避免大对象放全局状态。

但也不会盲目 memo，因为 memo 本身也有比较成本。只有在组件重、渲染频繁、props 稳定时才值得做。

## 15. 基础八股高频

### 问题：Vite 为什么比 Webpack 快？

**口述答案：**

开发阶段，Webpack 通常要先把整个项目打包成 bundle，再启动 dev server。项目越大，启动越慢。

Vite 利用浏览器原生 ES Module，开发阶段不需要整体打包，只在浏览器请求某个模块时按需转换。同时依赖预构建用 esbuild，速度比传统 JS 打包器快很多。

生产阶段 Vite 仍然会用 Rollup 打包，保证产物优化。

### 问题：pnpm 为什么节省磁盘？

**口述答案：**

pnpm 会把依赖存到全局内容寻址 store，项目里的 node_modules 通过硬链接和软链接引用这些依赖。同一个版本的包只会在磁盘上存一份。

同时 pnpm 的 node_modules 结构更严格，不会让项目随便访问没有声明的幽灵依赖。

### 问题：XSS 和 CSRF 有什么区别？

**口述答案：**

XSS 是攻击者把恶意脚本注入页面，脚本在用户浏览器执行，可能窃取 token、篡改页面或发请求。

CSRF 是利用用户已登录状态，诱导浏览器向目标网站发起非用户本意的请求。

防 XSS 要做输入输出转义、CSP、避免直接 innerHTML、token 不放可被 JS 读取的位置。防 CSRF 可以用 SameSite Cookie、CSRF Token、校验 Origin/Referer。

### 问题：localStorage 存 token 有什么风险？

**口述答案：**

localStorage 方便，但 XSS 后攻击者可以直接读取 token。更安全的方案是 HttpOnly Cookie，因为 JavaScript 读不到。

如果必须用 localStorage，就要加强 XSS 防护、缩短 token 有效期、配合 refresh token、退出登录清理 token，并且敏感操作要求二次校验。
